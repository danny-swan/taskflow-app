## Bug #1 (BLOCKING) — Локальная БД содержит "фантомные" пространства чужих аккаунтов и старых билдов

**Приоритет:** critical
**Область:** src/lib/sync/pull.ts, src/lib/db.ts, src/store/useStore.ts, sidebar

### Симптомы (со слов пользователя)
- В сайдбаре видно **два «Мои задачи»**: одно со старыми задачами (текущий аккаунт), второе — пустое, без статусов и задач.
- Также видны пространства другого аккаунта (`lebedevdo.one+test1@gmail.com`), причём во всех написано «вы — владелец».
- Кнопка «Удалить» на этих фантомных personal-пространствах недоступна (текст: «личное пространство нельзя удалить»).
- То же самое видно на втором аккаунте на том же ПК.

### Что реально на сервере (проверено SQL 14.07 20:57 MSK)
```
ws_dc3e4f8… personal owner=dolebedev.9@yandex.ru       (41 задача, 0 статусов)
ws_f82791… personal owner=dolebedev.9@gmail.com         (0/0)
ws_9ef5d9… personal owner=lebedevdo.one+test1@gmail.com (26/7)
ws_fc592c… personal owner=lebedevdo.one@gmail.com       (4/7)
```
Ни одного shared. Ни одного invite. Ни одного второго personal у любого аккаунта.

### Корневая причина
1. **Pull-без-подчистки:** `pullAll` кладёт в локальный SQLite ВСЕ ws-строки, которые прилетают под RLS текущего пользователя. Из-за багов истории (Причина B из отчёта PR #95) под RLS текущего юзера иногда прилетали чужие ws-строки через невалидное membership, оседали в локальной `workspaces`, и с тех пор оттуда не удаляются. `clearUserData` теперь сбрасывает указатели (fix #95), но **не чистит `workspaces`/`workspace_members`** — старые чужие строки остаются.
2. **Sidebar не фильтрует по membership текущего юзера:** список пространств строится из локальной таблицы `workspaces` целиком, без проверки «есть ли в `workspace_members` строка `(user_id = auth.uid(), workspace_id = this)`». Поэтому в сайдбаре у одного и того же локального SQLite видны пространства всех аккаунтов, которые когда-либо на нём логинились.
3. **Membership-строка "owner" отрисовывается статически:** UI помечает первый row в `workspace_members` (или строку с `role='owner'`) как "вы — владелец", не сверяясь с тем, совпадает ли `user_id` этой строки с текущим `auth.uid()`. Отсюда и «во всех написано вы — владелец» на пространствах, где ты вообще не член.
4. **«Второе Мои задачи» (то, что пустое)** — вероятно, ws_local, оставшийся от pre-fix v11 (или ws другого аккаунта после смены пользователя), которое `reconcilePersonalWorkspace` в #95 переставило указатели, но не удалило саму ws-строку.

### Ожидаемое поведение
- В сайдбаре текущего пользователя должны быть **только** пространства, где у него есть активная строка в `workspace_members` (не soft-deleted) с ролью viewer/editor/owner.
- Никаких фантомов чужих аккаунтов, никаких дублей.
- "вы — владелец" отображается **только** если membership.user_id === auth.uid() AND membership.role === 'owner'.

### Что чинить (предложение, но подтвердить с исполнителем)
- **Клиент:** при `pullAll` сначала собрать все `workspace_id`, которые прилетели с сервера в `sync_workspace_members` под текущим user_id как активные. Затем в локальном `workspaces`/`workspace_members` **удалить всё, чего нет в этом наборе** (или soft-delete). Опция: делать это только в первом pull после логина/смены аккаунта.
- **Клиент:** в `clearUserData` дополнительно `DELETE FROM workspaces`, `DELETE FROM workspace_members`, `DELETE FROM workspace_settings` (сейчас их не трогает, чтобы не потерять локальные метаданные — но именно это создаёт проблему).
- **Sidebar / query:** список пространств строить через JOIN на `workspace_members` c `user_id = <current>` and `deleted_at is null`.
- **UI:** признак "вы — владелец" считать из membership строки текущего пользователя, а не из первой попавшейся.

### Разовое лечение локальной БД пользователя
Пользователь дал разрешение почистить локальную БД. Проще всего: «Стереть все данные» → перезайти (после фикса это будет чисто). Либо руками через отладочный SQL: `DELETE FROM workspaces WHERE uuid NOT IN (SELECT workspace_id FROM workspace_members WHERE user_id = (SELECT value FROM settings WHERE key='bound_user_id'));`

## Bug #2 (BLOCKING) — Приглашение в shared workspace: RPC отклоняет владельца

**Приоритет:** critical
**Область:** src/lib/sync/workspace.ts + UI приглашений; RPC `invite_to_workspace` не при чём (сервер проверяет корректно).

### Симптомы
- Пользователь создал общее пространство.
- В участниках видит "вы — владелец".
- При попытке пригласить участника по TF-XXXXXX получает ошибку: **"только владелец может пригласить"**.

### Корневая причина (гипотеза, подтвердить чтением кода)
Есть три возможных корня — надо разбираться:

**(a) Общее пространство не долетело до сервера.** На сервере ноль shared. Значит клиент создал ws только локально (в SQLite), не запушил его на сервер, а `invite_to_workspace` (RPC на сервере) под RLS не находит такого workspace и/или не находит owner-membership текущего юзера. Возможные причины:
- `syncNow` не запустился между "создать пространство" и "пригласить";
- push упал, а UI не показал ошибку;
- ws создалось с `kind='shared'` только локально, но на сервер уходит с `kind='personal'` (баг маппинга);
- ws создалось с `owner_id != auth.uid()` из-за баги #1 (в UI "вы — владелец" на любом ws).

**(b) `has_workspace_role(workspace_id, uid, 'owner')` возвращает false** — либо потому что owner-membership не создался (триггер `ensure_owner_membership` в 0028 срабатывает только для personal? — проверить), либо потому что клиент шлёт неверный workspace_id.

**(c) UX-баг:** owner в сайдбаре нарисован для чужого workspace (см. Bug #1), пользователь физически не owner на сервере, RPC корректно отклоняет.

### Что чинить
- **Клиент:** прежде чем показывать форму приглашения, дождаться `syncNow` — убедиться что ws запушен и `has_workspace_role(..., 'owner')` реально true. Показывать ошибку "пространство ещё синхронизируется" вместо "только владелец может пригласить".
- **Показывать пользователю причину:** какой именно код вернул сервер (`42501` vs `22023` vs "не найден") — сейчас всё сваливается в один текст.
- **Проверить триггер создания shared:** есть ли автосоздание owner-membership при INSERT в `sync_workspaces` с `kind='shared'` (в 0028 надо перечитать). Если нет — клиент должен сам создать membership перед приглашением.

### Как воспроизвести
1. Залогиниться под `lebedevdo.one@gmail.com` (нужен pro/trial/lifetime).
2. Создать общее пространство.
3. Проверить SQL: `SELECT * FROM public.sync_workspaces WHERE kind='shared' AND owner_id=<uid>;` — должно быть.
4. Проверить: `SELECT * FROM public.sync_workspace_members WHERE workspace_id=<ws> AND user_id=<uid>;` — должна быть роль owner.
5. Если обе строки есть — вызвать `invite_to_workspace(<ws>, 'TF-XXXXXX', 'editor')` через `supabase.rpc` в консоли — должно вернуть invite, а не 42501.

## Bug #3 (major) — Личное пространство: логика "самое первое = стандартное, остальные удаляемы"

**Приоритет:** major (UX)
**Область:** UI (кнопка удаления), src/lib/sync/workspace.ts (правило "делать нельзя")

### Что просит пользователь (verbatim)
> Мы можем сделать так, чтобы самое первое пространство (личное), которое появляется при регистрации было стандартным и его удалить нельзя, а все новые которые создаются удалять можно.

### Текущее поведение
Кнопка удаления недоступна на любом personal workspace (текст: «личное пространство нельзя удалить»).

### Ожидаемое поведение
- Первое personal-пространство (созданное автоматически при регистрации триггером `ensure_personal_workspace`) — **неудаляемое** (защитный флаг, ID совпадает с `ws_<uid>`).
- Дополнительные personal-пространства (если пользователь создал вручную) — **можно удалять**.
- При удалении дополнительного personal → все задачи/статусы/теги внутри него удаляются каскадом (FK on delete cascade, 0030). Возможно, добавить UX-подтверждение "будут удалены N задач/M статусов".

### Что чинить
- **Модель:** признак «системное personal-пространство» = `id == computeWorkspaceId(owner_user_id)` (детерминированный ID для первого ws). Все остальные personal — не системные.
- **RLS/сервер:** политика DELETE на `sync_workspaces` — только для не-системных personal (или shared owner).
- **UI:** кнопка удаления активна для не-системных ws; текст ошибки: «Основное личное пространство удалить нельзя. Создайте новое личное пространство или переносите задачи в другое.»

### Дополнительный вопрос к пользователю (в новом треде уточнить)
- Нужна ли вообще возможность создавать несколько personal-пространств? Сейчас лимит `personal_limit=2` (миграция 0029). Возможно, "второе Мои задачи" появилось не как фантом, а как второй personal с лимитом 2. Проверить.

## Bug #4 (major) — Новые пространства должны создаваться с дефолтными статусами

**Приоритет:** major (UX + функциональность)
**Область:** триггер на сервере (`ensure_default_statuses`?) или клиент (при `createWorkspace`)

### Что просит пользователь (verbatim)
> надо чтобы в каждом новом пространстве по стандарту были сразу статусы по умолчанию, которые были изначально (Важно, Сегодня, В процессе, Взять в работу, Приостановлено, Выполнено, причём статус выполено технический и он не удаляется, такая логика было до того, как сделали пространства.)

### Текущее поведение
Новые пространства создаются пустыми: ни статусов, ни тегов. Задачи создать нельзя (нет куда положить).

### Ожидаемое поведение
При создании любого нового пространства (personal или shared) автоматически сеются 6 статусов в фиксированном порядке:
1. Важно
2. Сегодня
3. В процессе
4. Взять в работу
5. Приостановлено
6. Выполнено (технический, is_default=true, is_system=true, неудаляемый, конечное состояние)

**Флаг `is_system`/`is_terminal` на статусе "Выполнено":** удалить его через UI нельзя, изменить имя — возможно, да (уточнить). При удалении задачи с этим статусом — возможно, задача уходит в архив, не в бэклог (уточнить логику).

### Что чинить
- **Сервер:** триггер `ensure_default_statuses_on_workspace_insert` (после INSERT в `sync_workspaces`), который создаёт 6 статусов с заданными именами, sort_order, и метит "Выполнено" как `is_system=true` (если такого столбца нет — добавить).
- Триггер должен быть идемпотентным (если по каким-то причинам пересоздают — не дублировать).
- **Клиент:** после INSERT ws принять статусы через realtime/pull — они появятся автоматически. `ensureSeededIfEmpty` в клиенте оставить как fallback, но не как основной путь.

### Открытый вопрос
- Модель до пространств имела такую же логику "Выполнено = system"? Нужен ли столбец `is_system` в `sync_statuses`? Проверить историю схемы.
- Что делать с существующими 4 personal-workspace, где статусы уже есть (главный аккаунт) или отсутствуют (yandex)? Backfill-миграция?

## Bug #5 (major) — Роль editor не должна менять статусы или настройки пространства

**Приоритет:** major (изначально согласованное правило, уже упоминалось до Wave A)
**Область:** миграция 0035 (planned PR-d-01), RLS-политики + client guards

### Что просит пользователь (verbatim, из прошлых обсуждений)
> editor должен менять статус задач НО НЕ должен иметь доступа к настройкам пространства; настройки/статусы = owner-only (или owner/admin если появится admin роль)

### Текущее состояние (после миграции 0031)
```
sync_statuses / sync_tags / sync_task_templates:
  insert/update/delete = editor
sync_workspace_settings:
  select = viewer, insert/update/delete = owner
```
То есть editor может изменять справочник статусов (создавать/редактировать/удалять), хотя не должен.

### Ожидаемое поведение
- editor: полный CRUD только на `sync_tasks` и `sync_task_hold_periods` (и `sync_overdue_events`? — уточнить). Может менять `status_id` конкретной задачи, но не редактировать сам справочник статусов.
- editor: SELECT на `sync_statuses`, `sync_tags`, `sync_task_templates`, `sync_workspace_settings` (viewer-level).
- owner: полный CRUD на всём в пространстве.
- Отдельная роль `admin` не вводится в этой волне (можно оставить на будущее).

### Что чинить (миграция 0035)
- В миграции 0035 переопределить `_insert_ws_role/_update_ws_role/_delete_ws_role` для `sync_statuses`, `sync_tags`, `sync_task_templates` с `editor` → `owner`.
- Проверить, не станет ли это дырой в задаче-CRUD (задача ссылается на status_id и tag_id, но editor должен только СВЯЗЫВАТЬ существующие).
- В клиенте: спрятать UI создания/редактирования статусов/тегов у editor.

Если код клиента ожидает изменённый `updated_at` при обновлении строки — надо ставить его руками в приложении. Проверить в pull-cursor логике.
