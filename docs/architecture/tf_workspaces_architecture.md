## 1. Краткое описание текущей архитектуры (что уже есть)

Ниже — только факты из кода, без интерпретаций.

### 1.1 Однопользовательская offline-first модель
- Локально: SQLite (tauri-plugin-sql / sql.js). Локальная база **принадлежит одному пользователю** — колонки `user_id` в ней нет вовсе. Изоляция «свой vs чужой» — на уровне сессии Supabase, а не запроса.
- Облако: Supabase Postgres + Realtime. Все sync-таблицы имеют `user_id uuid NOT NULL` и RLS `user_id = auth.uid()` (4 политики: SELECT/INSERT/UPDATE/DELETE).
- В прод синхронизируется 6 sync-сущностей в фиксированном порядке (`PUSH_ORDER`, mappers.ts:490):
  `statuses → tags → tasks → task_templates → overdue_events → task_hold_periods`.

### 1.2 Sync — где всё склеивается
- **Push:** batch 50, `.upsert(payloads, {onConflict:'id'})`, ретраи с экспоненциальным backoff, permanent-error фильтр.
- **Pull:** курсор `updated_at` per-table (у `overdue_events` — по `id`, append-only), `WHERE user_id = <me> AND updated_at > cursor LIMIT 500`. LWW по `updated_at`. Сироты (например, задача без статуса) → `DeferRowError` замораживает курсор.
- **Outbox** — локальная `sync_outbox`, grain row-level по `(entity_table, entity_uuid)`, payload не хранится — push перечитывает свежую строку.
- **Realtime** — подписка на 6 sync-таблиц с фильтром `user_id=eq.<uid>`; сам payload не применяется, только триггерит debounced `pullAll(600мс)`.
- **Гейт:** весь sync-цикл платный — блокируется в `index.ts:215` через `isProOrTrial`, статус `'paywalled'`.

### 1.3 Задачи, статусы, теги, дедлайны — где источник истины
- **Задачи** (`sync_tasks`): `id`, `title`, `status_id text` (ссылка на статус, **без FK**), `tag_id text` (без FK), даты, `sort_order`, `archived`, sync-метаданные.
- **Статусы** (`sync_statuses`) — user-настраиваемая sync-сущность (не enum, не хардкод). Семантика через `behavior ∈ {top,middle,bottom,archive}` + флаги `is_technical/hidden/default_collapsed`. Сид: 7 статусов. **Часть логики завязана на ИМЯ** `'Приостановлено'` (для hold-периодов) — хрупкое место.
- **Теги** (`sync_tags`) — отдельная sync-сущность. Задача имеет максимум **один** тег (`tag_id` nullable, не junction).
- **Режим дедлайнов** (`overdue_mode ∈ {calendar,business}`) — в **локальной** таблице `settings`, **per-device, не синкается**. Используется в `Tasks.tsx` через `daysUntilDeadline(...)`. В Calendar/Stats не применяется.

### 1.4 Как страницы читают данные
Все страницы (`Tasks/Calendar/Stats/Dashboard`) читают данные **напрямую из `useStore`** (стор держит весь набор строк локального SQLite) + точечные SQL-агрегаты (например `overdueEventsByDate()`). Отдельного слоя доступа к данным по scope сейчас нет — фильтрация идёт только по видимости статуса (`archived/hidden/is_technical/behavior`) и `deleted_at IS NULL`. Это критично: **в клиенте нет ни одного места, где данные фильтруются по user_id, потому что этот вопрос решается «одна БД = один юзер»**.

### 1.5 User-модель (что уже готово к shared)
- `auth.users` (Supabase) + `profiles` (наша): `id uuid PK → auth.users.id`, `email`, `metadata jsonb`, + **`public_user_id text UNIQUE NOT NULL` формата `TF-XXXXXX`** (12.07.2026, миграция 0026), `nickname`, `avatar_variant`, `bio`.
- Триггер `profiles_guard_immutable` гарантирует, что `id` и `public_user_id` **неизменяемы**.
- `TF-XXXXXX` был спроектирован именно как «идентификатор для приглашений» — фундамент под shared уже заложен.
- Тарифы: таблица `user_entitlements`, клиент через `getEntitlement()/isProOrTrial()`. Sync — только для paid/trial.

### 1.6 12 инвариантов, которые нельзя ломать (сжато)
1. RLS `user_id = auth.uid()` на всех `sync_*`. 2. `user_id NOT NULL` + мапперы всегда его проставляют. 3. Контракт sync-строки (`id/updated_at/deleted_at/version/client_id`). 4. `PUSH_ORDER` и `DeferRowError`. 5. Ссылки status_id/tag_id/task_id — text-uuid без FK, резолвятся клиентом. 6. Soft-delete везде. 7. Realtime фильтр `user_id=eq`. 8. `profiles` вне sync + guard immutability. 9. Entitlement-гейт (`'paywalled'`). 10. `SELECT *` в fetchLocal мапперов vs явные списки колонок в pull applier'ах — риск рассинхрона. 11. `overdue_mode` per-device, не синкается. 12. Клиентские миграции инкрементны (`user_version=10`, только v11).

## 2. Что именно вы хотите: терминология и продуктовая интерпретация

Ваше описание вводит две сущности одновременно, важно их разделить чётко:

- **Пространство (Space / Workspace)** — верхний контейнер с собственными настройками: **свои** статусы, свои теги, свой режим дедлайнов, свой набор участников (для shared). У пространства есть тип: **личное** (только владелец) или **общее** (owner + участники).
- **Проект** — тактическая группировка задач *внутри* пространства (обычно без своих статусов/тегов).

**Вы фактически хотите Space**, а не Project (потому что вы хотите отдельные статусы/теги/режим дедлайнов, а это workspace-level, не project-level). Дальше используем термин «пространство».

Продуктовая формулировка тарифов, которую вы дали, тоже сдвигается: правильнее ограничивать **пространства**, а не «проекты»:
- Free: 2 личных пространства, 0 общих.
- Pro: до 7 пространств суммарно, из них любые могут быть общими.

## 3. Архитектурные варианты (после анализа кода, а не абстрактно)

### Вариант A. `workspace_id` в каждой sync-строке (multi-tenant в одной БД). ⭐

Локальная БД остаётся одна. В каждую sync-таблицу (`sync_tasks/statuses/tags/task_templates/overdue_events/task_hold_periods` + новые `sync_workspaces/workspace_members`) добавляется колонка `workspace_id text NOT NULL`. RLS расширяется: доступ = «row.workspace_id принадлежит workspace, в котором user_id есть в `workspace_members`». Realtime-фильтр меняется с `user_id=eq` на `workspace_id=in.(...)`. Клиент фильтрует все выборки по `currentWorkspaceId`.

**Плюсы (специфично для TaskFlow):**
- Sync-инфраструктура почти не меняется концептуально — те же outbox/push/pull/mappers, только с одной новой колонкой в контракте.
- Одна SQLite-БД, одна файловая точка бэкапа, одна цепочка клиентских миграций, один Realtime-канал.
- Shared работает естественно: коллега получает доступ через `workspace_members` — сервер сам отдаёт ему строки под его RLS.
- Кросс-workspace фичи (глобальный поиск, «мои задачи из всех пространств») тривиальны.

**Минусы:**
- Инвазивность: колонку надо добавить в **6 SQL-таблиц + 6 Cloud*Payload + 6 мапперов + 6 pull applier'ов + локальную схему + фильтры pull/realtime**. Симметричная работа, но пропущенное место = дыра в изоляции.
- Курсор pull `sync_last_pulled_<table>` — глобальный per-table. При нескольких пространствах с разным темпом курсор станет неточным. **Нужен курсор per (workspace, table)** — это отдельное изменение в модели курсоров.
- RLS усложняется: раньше `user_id=auth.uid()`, теперь — проверка членства через join / SECURITY DEFINER функцию `is_workspace_member(ws, uid)`.
- Все выборки в `useStore` и `useMemo` на страницах должны получить фильтр по `workspace_id` — без этого экраны покажут смешанные данные.

### Вариант B. Отдельная SQLite-БД (файл) на пространство

**Не работает в текущей архитектуре TaskFlow**, потому что:
- Sync построен на **одном** SQLite-соединении и одном outbox — придётся вести N подключений, N outbox'ов, N debounce-таймеров, N pull-циклов.
- Клиентские миграции применяются к одному файлу — при N файлах любое пропущенное = битое пространство.
- **Shared пространство физически негде разместить локально у нового участника** — он не может «получить» файл владельца. Придётся либо тянуть shared через отдельный не-SQLite слой, либо в первый момент шаринга генерировать новый файл — модель становится противоречивой.
- Кросс-workspace статистика/поиск становятся многофайловой болью.

Единственный сценарий, где B имел бы смысл — «полностью офлайн, никогда не синкается», но у вас всё уже завязано на Supabase и entitlements.

**Отбрасываем.**

### Вариант C. Гибрид: личные пространства = variant A, общие = отдельный tenant

Личные — просто `workspace_id` в существующих sync-таблицах. Для общих — параллельные таблицы `shared_tasks / shared_statuses / shared_tags` (с явным `workspace_id` и `workspace_members`), с независимым push/pull.

**Отбрасываем**, потому что:
- Удваиваем модель: 6 sync-таблиц → 12. Дублирование мапперов и applier'ов.
- Кросс-view (все мои задачи, включая общие) требует UNION по всему стеку клиента.
- Единственное преимущество — «случайно не смешать личное и общее» — решается корректной RLS и грамотным `workspace_id`, а не удвоением схемы.

### Вариант D. Два верхнеуровневых раздела «Личные» / «Общие», внутри — проекты

Именно та вторая опция, которую вы предложили. На практике это **вариант A с дополнительным атрибутом `kind ∈ {'personal','shared'}` на `workspaces` + отдельная страница-переключатель в UI**. Модель данных не меняется относительно A, меняется только группировка в UI (два раздела вместо плоского списка).

Это **не отдельная архитектура, это UX-вариация A**. Логично отдать это как продуктовое решение, обсудить в §7.

### Вывод по варианту
**Рекомендуется вариант A**, при этом UI может быть организован как в D (два раздела «Личные» / «Общие»), если вам так удобнее продуктово. Ниже все детали — про A.

## 4. Где физически хранятся общие пространства и как они попадают участникам

Это ваш ключевой вопрос — отвечаю отдельно, потому что в текущей архитектуре ответ не тривиален.

### 4.1 Сервер — единственный source of truth для shared
Общее пространство — это **строка в `sync_workspaces` в Supabase**, а не файл на устройстве владельца. С момента создания она существует независимо от того, онлайн ли создатель.

Кому «принадлежит»:
- **`owner_id uuid NOT NULL → auth.users(id)`** — тот, кто создал (для биллинга, каскадного удаления при удалении аккаунта, лимитов).
- **`workspace_members(workspace_id, user_id, role, joined_at)`** — фактический список тех, кто имеет доступ, включая владельца (запись `role='owner'`). Именно эта таблица используется RLS'ом.

### 4.2 Как shared попадает в локальную SQLite участника
Через тот же самый sync-цикл:
1. Пользователь-участник открывает приложение → `syncNow()` → `pullAll()`.
2. Pull запрашивает `sync_workspaces` и `sync_workspace_members` — RLS отдаёт ему только те workspace-строки, где он есть в `workspace_members`.
3. Затем pull `sync_tasks/statuses/tags/...` — RLS теперь возвращает не только `user_id=me`, но и «строки любых workspace, где я участник». Локально они сохраняются с их `workspace_id`.
4. Realtime подписан на `workspace_id=in.(<все мои ws>)` — изменения от коллег прилетают тем же путём, дебаунсенный pull применяет их.

**Ключевое:** shared-данные **не «живут на устройстве создателя»**. Они живут в Postgres, у создателя они — просто одно из его пространств локально, ровно так же, как у любого другого участника. Если создатель офлайн 3 недели, другие продолжают работать.

### 4.3 Что происходит, когда создатель удаляет аккаунт
- Через `ON DELETE CASCADE` из `auth.users` — все его личные пространства и данные исчезают.
- Общее пространство: политика по вашему выбору. Разумный дефолт — **если у общего пространства нет другого участника с ролью `admin`, оно удаляется**; иначе — передаётся старейшему по `joined_at` `admin`. В MVP можно жёстко: «нельзя удалить аккаунт, если ты единственный owner общих пространств», как делает GitHub.

### 4.4 Что если участник офлайн неделю
- Локально он видит слепок shared на момент последнего sync.
- Пишет офлайн → outbox копит операции с `workspace_id` того пространства.
- Онлайн → push → сервер применяет с LWW (`updated_at`). Если коллеги параллельно правили ту же задачу — «последний выигрывает» (см. §6.3).

### 4.5 «А если владелец офлайн и участник хочет что-то поменять?»
Работает через сервер — участник никак не зависит от устройства владельца. Это фундаментальное отличие от варианта B.

## 5. Permission-модель для shared

### 5.1 Что критично, а что нет (ваша гипотеза)
Вы предложили: критичные настройки (статусы, режим дедлайнов) — только owner/admin, теги — можно editor. Логика правильная, но нужно уточнить.

**Что действительно критично** (меняет поведение задач для всех участников):
- Статусы (набор, behavior, порядок) — да, критично. Смена behavior статуса меняет фильтры, архив, hold-логику.
- Режим дедлайнов (`overdue_mode`) — да, критично. Меняет расчёт просрочки у всех.
- Название/удаление пространства — да, критично.
- Управление участниками (добавить/удалить/сменить роль) — да, критично.
- Изменение тарифных настроек пространства (кто платит за него) — да.

**Что не критично** (не ломает поведение, только удобство):
- Теги (добавить/переименовать) — да, можно editor. Ничего не сломает, кроме визуального смысла тега.
- Шаблоны задач — можно editor.
- Sort order задач и статусов **внутри пространства** — можно editor.

### 5.2 Минимальная ролевая модель для MVP

**3 роли, не больше:**

| Роль | Задачи | Теги | Статусы | Настройки ws | Участники |
|------|--------|------|---------|--------------|-----------|
| **owner** | CRUD | CRUD | CRUD | CRUD | пригласить/удалить/сменить роль |
| **editor** | CRUD | CRUD | read-only | read-only | не может |
| **viewer** | read-only | read-only | read-only | read-only | не может |

Owner и «admin» на первом шаге объединены — второй admin появится, только если понадобится делегирование. Для MVP этого достаточно.

Почему **три, а не две** (owner/editor): без `viewer` неудобно показывать пространство коллегам, которые не должны в нём писать (например, менеджер смотрит статус задач команды). Viewer — самый безопасный уровень доступа, легко реализуется через RLS (SELECT-only).

### 5.3 Как это реализуется на сервере
- Роли — колонка `role text CHECK (role IN ('owner','editor','viewer'))` в `workspace_members`.
- RLS SELECT — «есть запись в `workspace_members` с моим `user_id`». Роль не проверяется, все три роли видят.
- RLS INSERT/UPDATE/DELETE — «есть запись с role IN ('owner','editor')» (viewer заблокирован).
- Для статусов/настроек — своя политика: «role='owner'».
- Клиентские тултипы `disabled + hint` — только для UX, не как барьер безопасности. Реализовано в PR-c-05 через хелперы `useCanEdit()` / `useIsViewer()` (`src/store/workspaceScope.ts`) и единый tooltip `ws_viewer_readonly_tooltip`; подробности — wave-c-plan.md §5.

**Best practice:** обернуть проверку членства и роли в **SECURITY DEFINER функцию** `has_workspace_role(ws uuid, uid uuid, min_role text)` с явным `search_path` — так RLS-политики читаются одинаково и оптимизируются планировщиком.

## 6. Влияние на sync, безопасность, риски

### 6.1 Что меняется в sync (по компонентам)
| Компонент | Изменение | Инвазивность |
|-----------|-----------|--------------|
| SQL sync-таблицы | `ADD COLUMN workspace_id text NOT NULL` + индексы + backfill | Средняя |
| RLS каждой sync-таблицы | Заменить `user_id=auth.uid()` на `is_workspace_member(row.workspace_id, auth.uid())` | Высокая (безопасность) |
| `sync_workspaces`, `sync_workspace_members` | Новые таблицы + RLS | Средняя |
| `Cloud*Payload` в mappers.ts | Добавить `workspace_id: string` | Низкая, но во всех 6 |
| `*ToCloudPayload(row, userId, clientId)` | Подтягивать `workspace_id` из локальной строки | Низкая |
| `pull.ts` applier'ы | Явно перечислить `workspace_id` в INSERT/UPDATE — **риск забыть** | Средняя (тестами) |
| Pull-курсор | Расширить с `<table>` до `<table>+<workspace_id>` в settings | Средняя |
| Realtime-фильтр | С `user_id=eq` → `workspace_id=in.(...)` + переподписка при добавлении ws | Средняя |
| `useStore` и все страницы | Все SELECT'ы фильтруются по `currentWorkspaceId` | Высокая (много точек) |
| Клиентские миграции | Новая v11: add columns + `workspaces`/`workspace_members` + backfill Personal | Средняя |
| Outbox | Не меняется (уже row-level) | Нулевая |
| Мапперы `resolveStatusUuid`/`resolveTagUuid` | Учитывать `workspace_id` при resolve (статус из того же ws) | Средняя |

### 6.2 Что критически может поломаться и как это тестировать
- **Утечка чужих данных** через забытый RLS-переход. Тест: pgTAP-сценарий с двумя тестовыми юзерами, каждый в своём workspace + один общий → каждый видит ровно то, что должен.
- **Пропущенный `workspace_id` в pull applier'е** — задача сохранится с `NULL workspace_id` → не попадёт ни в одно пространство UI. Тест: интеграционный (fake pull) для каждой из 6 таблиц.
- **Курсор pull ушёл вперёд, а участника только что добавили** — история старых задач ws не приедет к нему. Решение: при добавлении в ws сбрасывать курсоры per (ws, table) для этого клиента, чтобы полный пул сработал.
- **DeferRowError каскад:** задача из ws приезжает раньше, чем статусы этого ws (из-за курсоров). PUSH_ORDER регулирует push, но pull применяет строки в порядке приезда. Сейчас есть механизм отсрочки — надо убедиться, что он корректно работает с workspace_id.
- **Смешивание статусов между пространствами:** задача из ws-1 ссылается на `status_id` из ws-2. Клиентская валидация в мапперах должна её отклонить или resolve через ws-scoped-поиск.
- **Realtime-подписка на in-list**: Supabase ограничивает длину in-фильтра. Если у юзера >20-30 пространств — придётся дробить каналы. Для MVP лимит 7 → не проблема.

### 6.3 LWW и совместное редактирование
Ваш текущий LWW (`updated_at`) — «последний выигрывает» на уровне всей строки, без per-field merge. Для shared это означает: если Алиса и Боб одновременно редактируют одну задачу, победит тот, чей push долетел позже. Для MVP — приемлемо (это стандартная модель для большинства shared-todo продуктов), но зафиксируйте это как осознанное ограничение и, возможно, покажите юзеру индикатор «редактируется участником N» позже.

### 6.4 Риски по уровням

| Риск | Уровень | Что делать |
|------|---------|-----------|
| Утечка чужих данных из-за RLS-ошибки | **High** | pgTAP-сценарии с двумя юзерами до мержа. Никакого «на прод сначала посмотрим». |
| Пропущенная колонка в pull applier'е → NULL workspace_id | **High** | Юнит-тест на каждую таблицу + pgTAP на «после pull все строки имеют workspace_id NOT NULL». |
| Забытый фильтр `currentWorkspaceId` в UI → смешение данных | **Medium** | Централизованный store-хук `useWorkspaceScopedTasks()`, а не голый `useStore(tasks)`. Плюс dev-варнинг при чтении tasks без фильтра. |
| Realtime теряет события при переключении ws | **Medium** | Обязательный `pullAll()` при переключении ws (без веры в realtime). |
| Курсоры сходят с ума при добавлении в новый ws | **Medium** | Сбрасывать per-(ws,table) курсоры при первом появлении членства. |
| Backfill существующих юзеров ломается | **Medium** | Идемпотентная миграция, dry-run в staging, отсрочка `SET NOT NULL` до успешного backfill. |
| LWW-конфликты на shared | **Low-Medium** | Принять как ограничение MVP, задокументировать. Presence-индикатор — позже. |
| Хрупкая логика по имени `'Приостановлено'` в shared | **Low-Medium** | Пофиксить одновременно (перейти на behavior/флаг), а не после. |
| Хранилище/пути | **Low** | Ничего не меняется — один файл `taskflow.db` как сейчас. |
| Кросс-workspace лимиты (free/paid) | **Done (PR-5, `feat/ws-a-05-limits`)** | Реализовано: BEFORE INSERT триггер `enforce_workspace_limit()` на `sync_workspaces` + SECURITY DEFINER функция `get_workspace_limit(uid, kind)`. Free=2 (personal), free shared=0, paid (pro/trial/lifetime активный)=7 суммарно по всем kind. Ошибка `workspace_limit_exceeded` (P0001). Клиентский UX-гейт — отдельно, см. §7.4-факт. |
| Переиспользование `TF-XXXXXX` для приглашений | **Low** | Уже есть, только UI-инвайт. |

### 6.5 Безопасность как чек-лист
1. RLS-функция `has_workspace_role(ws,uid,min_role)` — SECURITY DEFINER, `SET search_path = public, pg_catalog`, EXECUTE отозван у anon.
2. INSERT в `sync_*` требует, чтобы у автора была роль ≥ editor в этом ws (не просто членство).
3. DELETE участника — только owner. UPDATE role — только owner. Отдельные политики.
4. Триггер на `sync_tasks` (BEFORE INSERT/UPDATE): проверяет, что `status_id`/`tag_id` принадлежат тому же `workspace_id`. Иначе — reject.
5. pgTAP: минимум 2 тестовых юзера, 3 сценария (personal, shared owner, shared viewer), 40+ assert'ов.

## 7. UX и продуктовая логика

### 7.1 Где переключать пространство
**Рекомендую: в шапке приложения**, левый верхний угол, как в Notion/Linear/Slack. Причина — переключение меняет содержимое **всех** страниц (Tasks, Calendar, Stats, Dashboard, Settings), значит переключатель должен быть выше уровнем, чем сами страницы.

Почему **не во вкладке «Задачи»** (ваша первая идея): статистика/календарь/dashboard в этом случае должны отдельно узнавать «какое пространство сейчас смотрит юзер», получается тройная привязка. Проще — один глобальный `currentWorkspaceId` в сторе + переключатель, который его меняет.

### 7.2 Как показать «Личные» / «Общие» (ваша идея D)
Логично, если пространств будет много (у Pro до 7). Реализация: в дропдауне переключателя — два раздела с заголовками:
```
▼ Мои задачи (личное)
─ ЛИЧНЫЕ ─────
  Мои задачи ✓
  Личные цели
─ ОБЩИЕ ──────
  Работа (owner)
  Домашние дела
─ ─────────────
  + Создать пространство
  ⚙ Настройки пространств
```
На уровне модели данных — одна колонка `kind ∈ {personal,shared}` в `workspaces`, никакой дополнительной архитектуры не нужно.

### 7.3 Настройки — вкладки и уровни
- **Настройки → Аккаунт** (уже есть, миграция 0026): профиль пользователя (nickname, avatar, bio, TF-ID). Остаётся global.
- **Настройки → Пространство** (новое): статусы, теги, режим дедлайнов, участники (для shared), название/удаление ws. Всё — привязано к **текущему** пространству.
- В настройках пространства — вкладки: `Общее | Статусы | Теги | Дедлайны | Участники (для shared)`.
- При создании нового пространства — статусы сеются из стандартного пресета (тот же `SEED_STATUSES`), теги — пустой набор, `overdue_mode` = 'calendar' по умолчанию. Всё редактируется потом.

### 7.4 Тарифные ограничения — куда их прикладывать
**Ограничивать пространства, не задачи и не проекты.** Обосновано:
- Задачи и так «неограниченны» в текущей модели.
- Проекта как отдельной сущности в вашей архитектуре нет (у вас Space, а не Space+Project).

Free:
- **2 личных** пространства.
- **0 общих** пространств (кнопка «Создать общее» disabled с тултипом «Обновите до Pro»).
- Не может быть **приглашён** в общее? — тут два подхода:
  - Строгий: free вообще не может участвовать в shared. Простой и логичный: shared = только между Pro-юзерами.
  - Мягкий: free может быть viewer в чужом shared, но не может писать. Даёт бесплатным юзерам «попробовать» shared.
  - **Рекомендую строгий на MVP** — проще RLS, проще биллинг, легче объяснить.

Pro:
- До **7 пространств** суммарно (личные + общие).
- Может создавать общие и приглашать участников.
- Общий workspace **принадлежит одному Pro-юзеру** (owner) и «занимает» слот у него, не у участников.

### 7.4-факт Реализация лимитов (PR-5, `feat/ws-a-05-limits`, факт vs анализ)

> Добавлено после факта реализации PR-5. Раздел 7.4 выше — исходный анализ/рекомендация; здесь — что реализовано на самом деле, с расхождениями.

**Совпадает с анализом:** ограничение накладывается на пространства (не на задачи/проекты), free=2 личных/0 общих, paid=7 суммарно, серверный триггер — источник правды, клиентская проверка — только UX.

**Отличается от анализа / уточнено:**
- Лимит для paid читается из реальной таблицы `user_entitlements` (поля `plan ∈ {pro,trial,lifetime}`, `valid_until`) — функция `get_workspace_limit(uid, workspace_kind)`, SECURITY DEFINER, а не абстрактный «entitlement» из §1.5.
- Триггер `enforce_workspace_limit()` (BEFORE INSERT на `sync_workspaces`) считает `count(*) where owner_id=... and deleted_at is null and id <> new.id`, сравнивает с `get_workspace_limit`. При превышении — `RAISE EXCEPTION 'workspace_limit_exceeded' USING ERRCODE='P0001'`.
- Гейт применяется только когда `auth.uid() = owner_id` (через `IS DISTINCT FROM`, корректно для NULL) — пропускает service_role/backfill/суперпользователя. Это уточнение появилось из-за pgTAP/тестового окружения, не было explicit в исходном анализе.
- Из-за порядка PUSH_ORDER (workspaces создаются раньше своих members-строк) честный подсчёт активных пространств требует SECURITY DEFINER, а не RLS-scoped count — иначе на момент вставки подсчёт может быть занижен.
- Имя триггера `enforce_workspace_limit` алфавитно сортируется после существующего `block_shared_workspaces` (буква e > b), поэтому попытка создать `kind='shared'` в Wave A всё ещё падает с check-constraint `23514` (регрессия PR-1), а не с P0001 — лимитная проверка до shared не доходит. Явно протестировано в pgTAP.
- Клиентское создание workspace в TaskFlow — офлайн-first (локальный INSERT + outbox), поэтому синхронный перехват серверной ошибки лимита в модалке создания невозможен. Реализован двухуровневый гейт:
  1. Проактивный UX-гейт в `CreateWorkspaceModal` (дизейбл кнопки + апселл-сообщение), основанный на клиентском подсчёте текущих ws и entitlement — предотвращает почти все случаи.
  2. Fallback на race-condition (два устройства одновременно создают ws у лимита): ошибка `workspace_limit_exceeded` прилетает через sync-цикл и показывается в sync-чипе `Sidebar` как понятное сообщение `ws_limit_sync_error`, а не сырой Postgres-текст.
- Миграция получила номер `0029` (не `0028`, как предполагалось в первичном анализе) — `0028` уже был занят другой миграцией к моменту PR-5.
- Тексты апселла (i18n, ru/en): `ws_limit_free_hint`, `ws_limit_paid_hint`, `ws_limit_sync_error` — выдержаны в стиле существующего `AccountSwitchGate` (амбер-стиль), см. `workspaces-plan.md` §3.8-факт для точных строк.

### 7.5 Влияние на страницы Tasks/Calendar/Stats/Dashboard
Всё продолжает работать так же, только с фильтром `workspace_id = currentWorkspaceId`. Пустой ws выглядит как пустой (сид статусов есть, задач нет). Смена ws — instant, потому что все данные уже в локальной БД.

### 7.6 Один нюанс: `overdue_mode`
Сейчас он per-device, не синкается (пункт 11 инвариантов). При переходе на пространства он становится **per-workspace, синкается**. Это осознанное breaking change настройки — но для юзера незаметное (personal ws наследует его текущее значение при миграции).

## 8. Рекомендуемая архитектура (итог)

**Модель: multi-tenant workspaces в одной локальной SQLite и одной облачной схеме, доступ через membership, роль как атрибут membership.**

Сжато:
- Новая сущность **`sync_workspaces`** (`id, owner_id, name, kind ∈ {personal,shared}, sync-мета`). Синкается.
- Новая сущность **`sync_workspace_members`** (`workspace_id, user_id, role ∈ {owner,editor,viewer}, joined_at, sync-мета`). Синкается.
- Новая сущность **`workspace_settings`** (`workspace_id PK, overdue_mode text, ... — то, что раньше было в client-side settings и становится per-ws`). Синкается.
- Колонка **`workspace_id text NOT NULL`** в 6 существующих sync-таблицах (`sync_tasks/statuses/tags/task_templates/overdue_events/task_hold_periods`) + локальных таблицах.
- RLS через SECURITY DEFINER функцию **`has_workspace_role(ws, uid, min_role)`**.
- Триггер **workspace-scoped integrity**: `sync_tasks.status_id`/`tag_id` должны принадлежать тому же `workspace_id`.
- UI: переключатель в шапке, вкладки настроек per-workspace, две группы «Личные» / «Общие» в дропдауне.
- Роли: **owner / editor / viewer** (3 роли, не больше).
- Тарифы: **лимит на количество пространств** (free 2 личных / Pro 7 суммарно + shared).
- LWW — принять как есть (осознанное MVP-ограничение).

## 9. Phased plan (безопасное внедрение)

Разделяю **на три волны** — важно не запихивать всё в один релиз, потому что shared требует отдельного цикла безопасности.

### Wave A — фундамент, personal-only (целится в v1.1.0)
**Задача:** ввести модель пространств, но без shared-функциональности. Ноль риска утечки чужих данных.

1. Серверная миграция `0027_workspaces_foundation`:
   - `sync_workspaces`, `sync_workspace_members`, `workspace_settings`.
   - Функция `has_workspace_role` (SECURITY DEFINER).
   - `ADD COLUMN workspace_id text NULL` во все 6 sync-таблиц.
   - **Backfill**: для каждого юзера создать `sync_workspaces` типа `personal` с именем «Мои задачи» + `workspace_members(owner)` + backfill `workspace_id` во всех его строках.
   - `SET NOT NULL` на `workspace_id`.
   - RLS переписать: `has_workspace_role(row.workspace_id, auth.uid(), 'viewer')` (пока все — owner своего personal, поведение не меняется).
   - Realtime publication + переключение фильтра на `workspace_id=in.(...)`.
2. Клиентская миграция v11: аналогично локально.
3. Мапперы: `workspace_id` в 6 Cloud*Payload + чтение из локальных строк.
4. Pull applier'ы: явно добавить `workspace_id` в INSERT/UPDATE каждой из 6 таблиц.
5. Курсор: расширить с `sync_last_pulled_<table>` до `sync_last_pulled_<workspace>_<table>` (нужна миграция settings, но она тоже уже есть).
6. Store: `currentWorkspaceId` + все SELECT'ы через фильтр. Централизованный хук `useCurrentWorkspaceTasks()`.
7. UI: переключатель в шапке, «Мои задачи» единственный пункт на первом этапе, кнопка «+ Создать» — открывает диалог «Личное пространство» (для free — до 2, для Pro — до 7).
8. Настройки → per-workspace вкладка. `overdue_mode` мигрирует из local settings в workspace_settings этого personal.
9. UI лимитов (free 2 личных, Pro 7 суммарно), серверный триггер-энфорсер на INSERT в `sync_workspaces`.
10. **Shared-функциональности пока нет**: колонка `kind` только `'personal'`, kind check-constraint не пускает `'shared'`.
11. pgTAP: 30+ assert'ов на новую RLS/backfill/triggers.
12. Ручное регрессионное тестирование: обычный юзерский флоу (создать задачу, календарь, статистика) работает как раньше.

Итог Wave A: юзеры получают возможность иметь несколько личных пространств. Модель данных готова к shared, но shared не открыт.

### Wave B — shared (целится в v1.2.0)
**Задача:** открыть общие пространства.

1. `sync_workspaces.kind` теперь принимает `'shared'`. Триггер разрешает создать shared только Pro-юзеру.
2. UI приглашения по `TF-XXXXXX` (публичный ID уже есть с миграции 0026): экран «Пригласить участника» в настройках workspace.
3. Sync `sync_workspace_members` уже работает — при add member RLS автоматически даёт коллеге доступ к строкам.
4. Клиент: при появлении нового членства в realtime — pull всего этого ws (курсоры per-ws сбрасываются).
5. UI ролей: owner/editor/viewer. Соответствующие RLS-политики.
6. Триггер workspace-scoped integrity на `sync_tasks` (status/tag из того же ws).
7. Пофиксить хрупкую логику по имени `'Приостановлено'`: перевести на `behavior='paused'` или флаг `is_hold=true` (одна миграция, backfill).
8. pgTAP: сценарии с двумя тестовыми юзерами, включая попытки утечки (юзер X пытается прочитать/записать в ws юзера Y — RLS должен блокировать).
9. Presence-индикатор «редактируется коллегой N» — необязательно в MVP, можно позже.

Итог Wave B: полноценные общие пространства с ролями, RLS, тарифами.

### Wave C — полировка (v1.3.0+)
- Кросс-workspace поиск / «все мои задачи».
- Кастомные роли и pipeline статусов между ws.
- Экспорт пространства.
- Presence (PR-c-01, доставлено), per-field merge для совместного редактирования,
  если LWW окажется недостаточным.
- Historical audit log задач (PR-c-03, доставлено) — см. ниже.

#### Модель данных: `sync_task_activity_log` (PR-c-03)

Первая **pull-only** sync-сущность и первый серверно-пишущий журнал в проекте.
Отличия от 6 базовых sync-таблиц (которые клиент и читает, и пишет через outbox):

- **Таблица** `sync_task_activity_log` (миграция 0034): `id uuid PK`, `task_id uuid`
  (**без FK** — переживает удаление задачи), `workspace_id text NOT NULL`
  (**FK → sync_workspaces CASCADE**), `user_id uuid`, `kind text` (CHECK на 9
  значений: created/status_changed/deadline_changed/title_changed/
  description_changed/deleted/restored/tag_added/tag_removed), `payload jsonb`,
  `created_at timestamptz`. Индексы по `(task_id, created_at DESC)` и
  `(workspace_id, created_at DESC)`.
- **Immutable append-only.** RLS: `SELECT` членам пространства через
  `has_workspace_role(workspace_id, auth.uid(), 'viewer')`; `INSERT/UPDATE/DELETE`
  для `authenticated` — deny. Пишет только `SECURITY DEFINER`-триггер
  `log_task_activity()` (`trg_log_task_activity` на `sync_tasks`), обходя
  insert-deny. Гейт **только для `kind='shared'`** пространств — personal не
  логируется.
- **Приоритет событий** (одно изменение = одна запись): deleted → restored →
  status_changed → deadline_changed → tag_* → title_changed → description_changed;
  INSERT → created.
- **Клиент — pull-only.** Отдельный `PULL_ORDER = [...PUSH_ORDER, ACTIVITY_LOG_SPEC]`;
  спека `toCloud` кидает (в outbox/push не попадает). Аппликатор INSERT-only
  (существующий uuid — пропуск). Курсор пула — `created_at` (нет `updated_at`).
  Локальное зеркало — SQLite v13 (`task_activity_log`). Realtime — таблица в
  `WATCHED_TABLES`.
- **UI** — сворачиваемая секция «История изменений» в `TaskModal`, только для
  shared; автор резолвится как в Presence/MembersTab (я/presence-ник/короткий id,
  **никогда email**). Общий лог всего пространства (PR-c-04) — read-only вкладка
  «История» в `WorkspaceSettings` поверх того же зеркала, видна всем ролям только
  в shared.

#### Realtime: presence-канал (PR-c-01)

Presence — **отдельный от `postgres_changes` транспорт**: тот же
`supabase.channel(...)`, но эфемерное состояние «кто в канале сейчас», которого
нет и не должно быть в БД. **0 backend/DDL.**

- **Канал** `presence-ws-<workspaceId>` с `config.presence.key = userId` (несколько
  вкладок одного юзера → один ключ). Поднимается **только** для `kind='shared'`;
  на personal-пространство не создаётся вовсе.
- **Эфемерность.** Состояние вне оффлайн-first `useStore`/SQLite/outbox — в
  отдельном сторе `usePresenceStore`. Не переживает закрытие вкладки; unsubscribe
  делает `untrack()` + `removeChannel()` + `clear()`, иначе «призрачные» юзеры
  висят до heartbeat-таймаута сервера.
- **Приватность.** В presence-meta уходит только публичный минимум профиля
  (`nickname`/`avatar_variant`/`public_user_id`) — **email не трекается никогда**.
  Тот же принцип own-row RLS, что в `MembersTab` и audit-log.
- **Жизненный цикл** — хук `useWorkspacePresence()` симметричен
  `resubscribeRealtime`: переподписка при смене `currentWorkspaceId`, снятие при
  уходе с shared.

#### Roles: viewer-polish (PR-c-05)

UI-слой поверх ролевой RLS (см. §5.3) — сервер остаётся источником истины,

## 11. Изменения после Wave C (живой лог архитектурных решений)

### 11.1 (15.07.2026) Bootstrap-RLS для upsert нового пространства — миграция 0037 (PR #103)

**Контекст:** при первой облачной синхронизации только что созданного пространства клиент получал 403/42501 (см. roadmap §5 F7, §7.8). Здесь — архитектурный вывод, который важно помнить при любых будущих правках RLS пространств.

**Ключевой инсайт (upsert = ВСЕ три политики):** клиентский push через PostgREST делает `upsert` с `Prefer: return=representation` → `INSERT ... ON CONFLICT DO UPDATE ... RETURNING *`. Такой запрос проверяет НЕ только INSERT-политику, но и **SELECT-политику** (для RETURNING) и UPDATE-политику (для ветки ON CONFLICT). Если любая из них = false — весь запрос падает с 42501. Поэтому пробы обычным `INSERT` (без RETURNING) НЕ воспроизводят баг — надо тестировать тем же путём, что и клиент (upsert+RETURNING).

**Решение — узкое bootstrap-окно (а не доступ по `owner_id` навсегда):**
- `is_workspace_bootstrap(ws text, uid uuid)` — `true`, когда `uid` = `owner_id` пространства **И** в пространстве ещё нет ни одной активной строки членства. Как только появляется первое членство — bootstrap выключается, доступ строго по `has_workspace_role`.
- `workspace_has_active_members(ws text)` — вспомогательная; обе функции STABLE SECURITY DEFINER, `search_path=public,pg_catalog`, REVOKE anon/public + GRANT authenticated.
- **Почему источник истины — ЧЛЕНСТВО, а не `owner_id`:** при self-leave (при наличии второго owner'a) колонка `sync_workspaces.owner_id` может остаться указывать на ушедшего. Доступ «всем по owner_id» дал бы утечку. Bootstrap гаснет при первом членстве → утечки нет.
- **Тонкость для `sync_workspaces`:** её SELECT/UPDATE-политики НЕЛЬЗЯ выразить через `is_workspace_bootstrap(id, uid)`, потому что эта SECURITY DEFINER-функция выполняет подзапрос и **не видит in-flight строку**, которую как раз возвращает RETURNING того же INSERT. Поэтому для `sync_workspaces` условие вписано **инлайн**: `owner_id = auth.uid() AND deleted_at IS NULL AND NOT workspace_has_active_members(id)`. Для дочерних таблиц (ссылаются на `workspace_id`, а не на собственную in-flight строку) функция `is_workspace_bootstrap(workspace_id, uid)` работает корректно.

**Обновлённые ветки политик (к §6.5 чек-лист):**
- `sync_workspaces`: SELECT/UPDATE += инлайн-bootstrap (выше).
- дочерние (`sync_tasks`, `sync_statuses`, `sync_tags`, `sync_task_templates`, `sync_task_hold_periods`, `sync_overdue_events`) + `sync_workspace_settings` + `sync_task_activity_log`(SELECT): += `OR is_workspace_bootstrap(workspace_id, auth.uid())`.
- `sync_workspace_members`: SELECT += bootstrap ИЛИ own-row (own-row покрывает RETURNING собственной bootstrap-строки владельца); INSERT членства владельцем — как в 0027/0031 (`owns_workspace`).
- `sync_task_activity_log`: INSERT/UPDATE/DELETE остаются denied (серверный лог).
- Семантика ролей editor/viewer/owner (0031/0035) НЕ меняется — добавлена только bootstrap-альтернатива, живущая до первого членства.

**Клиентская часть:** миграция v14 разово сбрасывает outbox-строки, застрявшие из-за того, что `isPermanentError` (`src/lib/sync/push.ts`) считал 403/RLS постоянной ошибкой.

**Проверка:** pgTAP `18_upsert_returning_bootstrap_test.sql` (RED→GREEN), весь набор 18/18 зелёный; прод-SQL-пробы под ролью владельца 15.07.2026. Миграция 0037 применена на прод.

### 11.2 (21.07.2026) Двухфазный pull: приглашённый видит чужое пространство — клиентский фикс (PR #103, коммит `bc50b2d`)

**Контекст:** после фикса 403 (§11.1) оставался P0: приглашённый принимал приглашение (`accept_invite` 200, членство на сервере создавалось), но чужое пространство не появлялось (roadmap §5 F8, §7.9).

**Ключевой инсайт (клиент НЕ должен дублировать RLS фильтром по `user_id`):** серверные SELECT-политики всех `sync_*`-таблиц пространства скоупятся по **ЧЛЕНСТВУ** (`has_workspace_role(...,'viewer')`), а не по `user_id`. В shared-пространстве строки принадлежат ВЛАДЕЛЬЦУ (`user_id`=other), поэтому клиентский `.eq('user_id', me)` отсекал то, что сервер уже корректно отдавал. **Правило:** клиентский pull data-таблиц пространства должен скоупиться по `workspace_id` (по членству), а не по `user_id`.

**Курица-и-яйцо (chicken-and-egg):** набор пространств для pull (`listWorkspaceIds`) строился из локальной таблицы `workspaces`, а чужой ws попадал туда только через pull `sync_workspaces` — который сам был отфильтрован по `user_id`. Разорвано двухфазным pull.

**Решение — двухфазный pull (только клиент, 0 DDL):**
- **Фаза 1:** `sync_workspace_members` тянется по `.eq('user_id', me)` (это единственный легитимный вход в набор — свои строки членства в любом ws).
- **Пересчёт набора:** `listMembershipWorkspaceIds(userId)` — набор `workspace_id` из СВЕЖЕПОДТЯНУТОГО членства (не из `workspaces`) + всегда personal `computeWorkspaceId`.
- **Фаза 2:** `sync_workspaces` (скоуп `id` — у неё PK == ws-id, колонки `workspace_id` нет) и все data-таблицы (скоуп `workspace_id`) тянутся по набору пространств. `pullScope` (`mappers.ts`) расширен значением `'id'`.

**Тонкость — per-ws курсоры:** каждый ws в фазе 2 тянется СО СВОИМ курсором. Если бы был общий курсор «ведущего» ws (personal), то свежепринятый shared-ws со «старыми» `updated_at` (старше lastPulled personal) отсёкся бы `> cursor` и никогда не подтянулся. Новый ws → курсора нет → тянется целиком с нуля.

**Что НЕ меняли:** RLS и миграции не трогали (баг клиентский, сервер корректен). `listWorkspaceIds`/realtime/`loadWorkspaces` (сайдбар) оставлены как есть — после фазы 2 строка чужого ws уже в локальной `workspaces`, и они подхватывают её штатно. `prunePhantomWorkspaces` уже строил allow-list по членству — не мешает.

**Проверка:** vitest `pull.twophase.test.ts` (RED→GREEN, мок PostgREST с RLS-семантикой) + весь набор 556/556, `tsc -b`+`vite build`. Подтверждено дифференциальной прод-пробой: сервер по RLS отдаёт 1 ws + 3 задачи приглашённому, клиентский `user_id`-фильтр → 0.
