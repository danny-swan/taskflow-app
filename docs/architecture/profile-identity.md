# Профиль: модель идентификаторов (миграция 0026)

Кастомизация профиля вводит в `public.profiles` разделение между внутренним и
публичным идентификатором пользователя плюс косметические поля.

## Идентификаторы

| Поле             | Тип           | Роль                                                                                          | Показывается юзеру | Изменяемость                          |
|------------------|---------------|-----------------------------------------------------------------------------------------------|--------------------|----------------------------------------|
| `id`             | `uuid` (PK)   | **Внутренний ID**, равен `auth.users.id`. Связность/логика: FK, RLS own-row, join'ы.          | Нет                | Неизменяем (guard-триггер)             |
| `public_user_id` | `text UNIQUE` | **Публичный ID** формата `TF-XXXXXX`. Юзер сообщает его другим — поиск / будущее добавление в друзья. | Да                 | Неизменяем после присвоения (guard)    |

`public_user_id` генерируется из алфавита `ABCDEFGHJKMNPQRSTUVWXYZ23456789`
(без визуально похожих `I L O 0 1`), 6 символов → префикс `TF-`. Присваивается:

- новым пользователям — в триггере `handle_new_user()` при регистрации;
- существующим — backfill'ом в миграции 0026 (после чего колонка `SET NOT NULL`).

Уникальность подбирается с retry (до 10 попыток) функцией
`assign_public_user_id()`.

## Косметические поля

| Поле             | Тип        | Ограничение                        |
|------------------|------------|------------------------------------|
| `nickname`       | `text`     | `char_length ≤ 32`, NULL допустим  |
| `avatar_variant` | `smallint` | `1..8`, NOT NULL, default `1`      |
| `bio`            | `text`     | `char_length ≤ 160`, NULL допустим |

Это **дополнение** профиля, а не замена идентификатора: ник не уникален, аватар —
один из 8 встроенных SVG (индекс ↔ `avatar_variant`).

## Guard неизменяемости

У роли `authenticated` есть `UPDATE ON profiles` (RLS own-row), поэтому
BEFORE-UPDATE триггер `profiles_guard_immutable` молча возвращает старые значения
`id` и `public_user_id`, если их пытаются переписать. Легитимные UPDATE
(`nickname` / `avatar_variant` / `bio`) и смена email не блокируются. Триггер
отделён от `set_updated_at`.

## Sync

`profiles` **не участвует** в клиентском sync-цикле (`outbox → push → pull`).
Профиль читается/пишется отдельным Supabase-запросом (`src/lib/profile.ts`),
поэтому новые поля не влияют на upsert/conflict-flow таблиц `sync_*`.
