# Supabase Auth hardening (ops)

Оперативная инструкция по настройкам Supabase Auth, которые нельзя выставить
миграцией или кодом — только через Dashboard. Держим здесь, чтобы шаг не терялся
между релизами.

## N16 — Leaked password protection

**Что:** Supabase Auth умеет сверять пароль пользователя с базой утёкших паролей
(HaveIBeenPwned, k-anonymity — сам пароль наружу не уходит, отправляется только
префикс SHA-1 хэша). При регистрации/смене пароля скомпрометированные пароли
отклоняются.

**Почему:** защита от credential stuffing — пользователи часто переиспользуют
пароли, попавшие в публичные дампы. Это чисто дашбордная настройка Supabase Auth,
её нет в `config.toml` проекта и её нельзя включить миграцией.

**Как включить (требует ручного действия в Supabase Dashboard):**

1. Открыть проект в [Supabase Dashboard](https://supabase.com/dashboard).
2. **Authentication → Providers → Email** (либо **Authentication → Policies /
   Password**, в зависимости от версии дашборда).
3. Найти пункт **Leaked password protection** (в некоторых версиях —
   «Password strength / Prevent use of compromised passwords»).
4. Включить тумблер, сохранить.

**Проверка:** попробовать сменить пароль на заведомо утёкший (например
`Password123!`) — Supabase должен вернуть ошибку о скомпрометированном пароле.

**Область действия:** влияет только на новые регистрации и смены пароля; уже
сохранённые пароли не инвалидируются.

> ⚠️ Это действие выполняется вручную в Dashboard и не покрывается CI/миграциями.
> Отметить как выполненное в чек-листе релиза после включения.
