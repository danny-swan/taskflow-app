# 0004. Rate limiting — table-based счётчик в Postgres

- Статус: accepted
- Дата: 2026-07-11
- Связано: находка N13, Wave 4 PR-B, ветка `wave-4b-rate-limit`, миграция
  `0024_wave4_rate_limits.sql`, модуль `_shared/rate-limit.ts`. Соседние решения:
  [ADR 0002](0002-get-users-emails-internal-admin-gate.md) (доверенные заголовки),
  N8 (best-effort IP в `payment-webhook`).

## Контекст

Публичные edge-эндпоинты (`create-payment`, `start-trial`, `payment-webhook`)
до Wave 4 не имели ограничения частоты запросов. Это открывало флуд:
многократное создание платежей/чеков ЮKassa (каждый вызов — реальный чек НПД в
ФНС), брутфорс trial-активаций, заваливание webhook-обработчика (тяжёлый
dual-verify GET /v3/payments + запись в БД на каждый запрос).

Нужен лимитер, который:

- работает поверх **stateless, многоинстансных** Deno edge-функций (счётчик
  должен быть общим для всех инстансов и переживать холодный старт);
- не требует новой инфраструктуры (у проекта нет Redis/очередей);
- атомарно инкрементит счётчик под конкурентной нагрузкой.

## Решение

Общий счётчик в Postgres: таблица `public.rate_limits (key, window_start,
count, expires_at)` + функция `public.check_rate_limit(key, max, window_sec)`.

### Почему table-based, а не альтернативы

- **In-memory (Map в процессе функции)** — не подходит: каждый инстанс edge-
  рантайма держит свою память, счётчик не разделяется и обнуляется при cold
  start. Лимит был бы фиктивным.
- **Redis / Upstash** — надёжный отраслевой выбор, но требует поднять и
  оплачивать отдельный сервис, хранить ещё один секрет, добавить зависимость в
  рантайм. Для текущего масштаба это избыточно.
- **Postgres, который уже есть** — единый источник истины для всех инстансов,
  транзакционная атомарность из коробки, ноль новой инфраструктуры.

### Атомарность через INSERT ... ON CONFLICT

`check_rate_limit` делает один UPSERT: `INSERT ... ON CONFLICT (key) DO UPDATE`.
`ON CONFLICT DO UPDATE` берёт row-lock на конфликтующую строку, поэтому
конкурентные вызовы для одного ключа сериализуются — гонки инкремента нет.
Логика окна (fixed window) вычисляется прямо в `DO UPDATE`:

- нет строки → создаём окно, `count = 1`;
- окно истекло (`expires_at < now()`) → сброс: `count = 1`, новые
  `window_start`/`expires_at`;
- окно активно → `count + 1`, границы окна не двигаем.

Функция возвращает `allowed` и `retry_after` (секунды до конца окна при
превышении). `SECURITY DEFINER` + `SET search_path = public, pg_temp` (требование
N18); EXECUTE выдан только `service_role`.

### fail-open trade-off

Если RPC вернул ошибку (БД недоступна, таймаут) — `checkRateLimit` **пропускает**
запрос (`allowed: true`) и логирует ошибку. Осознанный выбор: rate-limiter —
вспомогательная защита, и уронить из-за его сбоя реальные платежи хуже, чем
на время потерять ограничение частоты. Симметрично fail-open в
[ADR 0003](0003-renewal-idempotency-guard.md).

### Доверенные IP-заголовки (связь с N8)

Ключ per-IP строится из `getClientIp`, который читает **только**
`cf-connecting-ip` → `x-real-ip` (заголовки ставит edge-runtime/Cloudflare перед
функцией). `x-forwarded-for` НЕ используется — он тривиально подделывается
клиентом, и лимит по нему обходился бы сменой заголовка (та же логика, что в N8
для `payment-webhook`).

### Cleanup через pg_cron

Истёкшие строки удаляет job `rate-limits-cleanup` (`*/5 * * * *`,
`DELETE ... WHERE expires_at < now()`). Создаётся идемпотентно (unschedule +
schedule) и обёрнут в guard по `pg_available_extensions` — на vanilla Postgres в
CI (где pg_cron нет) блок становится no-op, миграция проходит db-tests.

### Выбранные лимиты

| Эндпоинт          | per user      | per IP        | Обоснование |
|-------------------|---------------|---------------|-------------|
| `create-payment`  | 10 / min      | 30 / min      | Платёж/чек НПД — дорогая операция; человеку 10/мин с запасом хватает, IP-лимит выше (NAT/офис). |
| `start-trial`     | 3 / hour      | 5 / hour      | Trial активируется один раз на аккаунт — частые попытки подозрительны. |
| `payment-webhook` | —             | 60 / min      | User нет (webhook ЮKassa). ЮKassa шлёт с ограниченного пула IP; 60/мин покрывает легитимные пачки, но отсекает флуд. Проверка — ПОСЛЕ базовой валидации payload, ДО тяжёлого dual-verify. |

Для `create-payment`/`start-trial` проверяются оба лимита (user и IP); превышение
любого → `429` с заголовком `Retry-After`.

## Последствия

- Плюс: единый переиспользуемый модуль `_shared/rate-limit.ts`, ноль новой
  инфраструктуры, атомарность из коробки, тесты (pgTAP + Deno).
- Минус: небольшой оверхед — один RPC round-trip к БД на защищённый запрос
  (для платежей приемлемо). Fixed-window допускает всплеск до 2× на стыке окон —
  для этих эндпоинтов не критично.
- Порог сработавшего лимита возвращает `429 { error: 'rate_limited', retry_after }`
  — клиент может показать понятное сообщение и повторить позже.
