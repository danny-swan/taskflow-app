# 0004. Rate limiting — счётчики в таблице Postgres + атомарный SECURITY DEFINER RPC

- Статус: accepted
- Дата: 2026-07-10
- Связано: находка N13, ветка `wave-4-hardening`, миграция `0024_rate_limits.sql`, модуль `_shared/rate-limit.ts`

## Контекст

Публичные и платёжные edge-функции (`create-payment`, `start-trial`,
`payment-webhook`) не имели throttling. Аутентифицированный или анонимный
источник мог засыпать их запросами: спамить создание платежей в ЮKassa,
перебирать trial-активации, флудить вебхук-endpoint. Нужен лимит частоты запросов
на юзера и на IP.

Edge-функции Supabase stateless и масштабируются горизонтально — несколько
инстансов обслуживают запросы параллельно. Значит счётчик должен жить в общем
хранилище, а инкремент — быть атомарным, иначе параллельные запросы гоняются за
счётчиком (race → лимит протекает).

## Рассмотренные варианты

1. **In-memory на инстанс.** Просто, но бесполезно: каждый инстанс считает своё,
   общий лимит не соблюдается. Отклонено.
2. **Redis / Upstash.** Классический выбор для rate limiting (атомарный INCR + TTL).
   Но это новая внешняя зависимость, отдельный сервис, секреты, точка отказа и
   счёт. Для нашего трафика — избыточно. Отклонено.
3. **Таблица в Postgres + атомарный upsert.** Postgres у нас уже есть,
   `INSERT ... ON CONFLICT DO UPDATE ... RETURNING` атомарен в пределах строки,
   нагрузка тривиальна (один upsert на запрос). Выбрано.

## Решение

Фиксированное окно (fixed window) на ключ в таблице `public.rate_limits`:

- `key text PRIMARY KEY` — вида `${fn}:user:${userId}` или `${fn}:ip:${realIp}`.
- `window_start`, `count`, `expires_at` — состояние текущего окна.

Инкремент — единственный атомарный `INSERT ... ON CONFLICT (key) DO UPDATE ...
RETURNING` внутри SECURITY DEFINER функции `public.rate_limit_hit(p_key, p_max,
p_window_seconds)`. Если окно истекло (`expires_at <= now()`), тем же upsert-ом
счётчик сбрасывается в 1 и окно открывается заново; иначе `count + 1`. Функция
возвращает `(allowed, retry_after)`: `allowed=false`, когда после инкремента
`count > p_max`.

- Таблица заперта: `ENABLE ROW LEVEL SECURITY` + `REVOKE ALL FROM PUBLIC`. Прямого
  доступа у `anon`/`authenticated` нет. Пишется только через RPC (owner=postgres)
  и `service_role` (BYPASSRLS).
- `EXECUTE` на `rate_limit_hit` выдан только `service_role` — edge-функции зовут
  RPC под service-ключом.
- Клиентский хелпер `_shared/rate-limit.ts` дёргает RPC. Реальный IP берётся
  ТОЛЬКО из `cf-connecting-ip` / `x-real-ip`; `x-forwarded-for` НЕ используется —
  он тривиально подделывается (см. N8), иначе лимит по IP обходится подстановкой
  заголовка.
- Очистка протухших строк — `pg_cron` каждые 5 минут
  (`DELETE ... WHERE expires_at < now()`), под guard'ом доступности расширения
  (на vanilla-Postgres в CI пропускается, как в 0015/0018/0019/0023).

Лимиты (per-user + per-IP, первый сработавший блокирует):

| Функция | На юзера | На IP |
|---|---|---|
| `create-payment` | 10 / мин | 30 / мин |
| `start-trial` | 3 / час | 5 / час |
| `payment-webhook` | — (анонимный) | 60 / мин |

При блокировке — HTTP `429` с заголовком `Retry-After`.

**Fail-open по осознанному выбору.** Если RPC/сеть упали, `checkRateLimit`
возвращает `allowed=true` и запрос пропускается. Rate limiting — это слой защиты,
а не критичный путь: ложный отказ платежа хуже пропущенного лимита. Для вебхука
аутентичность всё равно гарантирует dual-verify, а не rate limit.

## Последствия

Плюсы: общий счётчик без новой инфраструктуры; атомарность на уровне строки без
внешних блокировок; таблица недоступна напрямую клиентам; чистые хелперы покрыты
Deno-тестами без сети (мок RpcClient), а SQL — pgTAP (`05_wave4_test.sql`).

Минусы: fixed window допускает всплеск до 2×`max` на стыке окон (запрос в конце
одного окна + запрос в начале следующего) — для нашей модели угроз приемлемо
(защита от флуда, не точный биллинг). Требует `pg_cron` для очистки; без него
таблица растёт (строки протухают логически, но физически не удаляются — можно
чистить вручную). Fail-open означает, что при недоступности Postgres лимит
временно не действует.

Применение на прод: миграция `0024_rate_limits.sql` + редеплой `create-payment`,
`start-trial`, `payment-webhook` (каждая тянет `../_shared/rate-limit.ts`). На
момент записи ADR не применено — код в `wave-4-hardening`.
