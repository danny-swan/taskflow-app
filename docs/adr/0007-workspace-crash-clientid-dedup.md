# 0007. Пространства — стабильная ссылка в zustand-селекторе, client_id при push = текущее устройство, самолечение дубля personal

- Статус: accepted
- Дата: 2026-07-22
- Связано: находка F13 (эпик «Пространства после смены аккаунта/переустановки»), ветка `feat/workspaces`, [ADR 0005](0005-shared-workspaces.md); только клиент, БЕЗ миграций

## Контекст

Симптом (баг, аккаунт `lebedevdo.one+test1@gmail.com`, свежий билд): при открытии
задачи в общем (shared) пространстве появлялось окно ошибки (AppErrorBoundary);
после «Перезагрузить» — пропадали общие пространства, появлялся задублированный
пустой «Мои задачи», часть задач личного пространства не отображалась, и висела
красная ошибка sync: `insert or update on table "sync_workspaces" violates foreign
key constraint "sync_workspaces_client_id_fkey"`.

Диагностика (код-анализ + прод-пробы под ROLLBACK на `sejpmzrmtgcvevukggkx`,
детерминированный vitest-репро) выделила ОДИН корневой краш (A) и два его
следствия (B, C):

- **A (корень, краш).** `TaskActivityLog` рендерится ТОЛЬКО в shared-пространстве
  (`TaskModal.tsx`, IIFE `isShared`). Хук `useTaskActivity` (`useTaskActivityStore.ts`)
  в селекторе для пустого журнала возвращал ИНЛАЙН-литерал `[]`:
  `s.byTask[taskUuid] ?? []`. Для zustand на `useSyncExternalStore` новая ссылка
  `[]` при каждом вызове снапшота = «состояние изменилось» → форс ре-рендера →
  снова новый `[]` → бесконечный цикл → React `Maximum update depth exceeded` →
  ловит AppErrorBoundary. Соседний `useWorkspaceActivity` в этом же файле уже
  использовал стабильную константу `EMPTY_RECORDS` — в task-хуке её забыли. Замысел
  по `wave-c-plan.md` §PR-c-03: «taskUuid=null → пустой результат, запросов нет»
  (то есть тихий пустой лог, без исключений) — то есть краш это регрессия
  реализации, а не задуманное поведение.
- **B (следствие, дубль personal).** На проде у пользователя РОВНО одна
  non-deleted personal-ws с детерминированным `ws_<uid>` (проверено). Дубль пустого
  «Мои задачи» — ЛОКАЛЬНЫЙ артефакт: осиротевшая personal-строка от ПРОШЛОГО
  аккаунта (`ws_<olduid>`), оставшаяся в SQLite после смены аккаунтов и аварийного
  reload (краш A прерывал загрузку/reconcile на середине).
- **C (следствие, FK sync).** FK `sync_workspaces.client_id → sync_devices(id)`.
  Мапперы push слали `client_id: row.client_id ?? clientId`. После смены аккаунта
  (или pull с другого устройства) в локальной строке оседал `client_id` СТОРОННЕГО
  устройства, которого нет в `sync_devices` текущего контекста → `23503`. При этом
  `push.ts isForeignKeyViolation` трактует ЛЮБОЙ 23503 как «ждём родительский
  workspace» → бесконечный ретрай → залипшая «ошибка sync». Прод-проба под ROLLBACK
  воспроизвела FK. У аккаунта накопился 21 device (новый `client_id` на каждую
  установку/переустановку — он в SQLite `settings`, генерится на свежей БД).

## Решение (три точечных клиентских фикса, один PR, без DDL)

1. **A — стабильная ссылка пустого журнала.** В `useTaskActivity`
   (`src/store/useTaskActivityStore.ts`) заменить инлайн `[]` на общую константу
   `EMPTY_RECORDS` (та же, что уже используется в `useWorkspaceActivity`):
   `s.byTask[taskUuid] ?? EMPTY_RECORDS`, и для null-ветки тоже `EMPTY_RECORDS`.
   Устраняет петлю ре-рендера → краш. Общее правило: **селектор zustand для
   «пустого» результата ОБЯЗАН возвращать стабильную ссылку**, иначе
   `useSyncExternalStore` уходит в бесконечный ре-рендер.

2. **C — при push `client_id` ВСЕГДА текущее устройство.** Во всех девяти
   `*ToCloudPayload` мапперах (`src/lib/sync/mappers.ts`) заменить
   `client_id: row.client_id ?? clientId` на `client_id: clientId`. Семантика
   `client_id` в облачной строке — «устройство, отправившее/последним изменившее
   строку»; при push строка физически уходит с ТЕКУЩЕГО устройства, которое
   гарантированно зарегистрировано в `sync_devices` (`ensureDeviceRegistered` перед
   push). Протухший `row.client_id` из pull больше не утекает обратно → FK 23503 по
   `client_id` не воспроизводится. `push.ts` НЕ меняем: раз `client_id` всегда
   валиден, оставшаяся трактовка 23503 как «ждём родителя» корректна для
   единственного оставшегося сценария FK — `workspace_id` (родитель ещё не долетел).

3. **B — самолечение дубля personal.** В `reconcilePersonalWorkspace`
   (`src/lib/sync/workspace.ts`) добавить `dedupePersonalWorkspaces(userId, target)`:
   soft-delete всех локальных `kind='personal'` строк, кроме канонической `target`
   (`ws_<uid>`), у которых у текущего пользователя НЕТ живого членства (чужой мусор
   от прошлых аккаунтов). Гасим ТОЛЬКО локально (зеркало), **БЕЗ `enqueueOutbox`** —
   это чужие строки под другим `user_id`, их нельзя пушить как удаление на сервер.
   Shared-ws не трогаем никогда. Строго идемпотентно (`WHERE deleted_at IS NULL`).

Схема БД не менялась → миграции/ERD не нужны.

## Последствия

Плюсы: краш при открытии задачи в shared закрыт (первопричина); дубль personal
самолечится при следующем sync/reconcile без переустановки; залипшая «ошибка sync»
по `client_id` больше не появляется (в т.ч. на базах со следами смены аккаунтов).
Все три — только клиент, без риска для прод-схемы.

Минусы/риски: (1) правило «стабильная ссылка в селекторе» негласное — легко снова
нарушить в новом хук��; закреплено регресс-тестом на идентичность ссылки.
(2) `client_id = clientId` теряет исходное устройство-автор в облачной строке при
ре-push, но это поле — телеметрия «последний писатель», не источник истины
конфликтов (LWW идёт по `version`/`updated_at`), поэтому потери смысла нет.

Верификация: vitest — регресс `useTaskActivity.selector.test.tsx` (стабильность
ссылки пустого журнала, 2 кейса), `workspaces-sync.test.ts` += кейс Bug B (гасит
осиротевшую personal, не трогает shared, не пишет в outbox, идемпотентно) и кейс
Bug C (маппер игнорирует протухший `row.client_id`, ставит текущий); существующие
`useTaskActivityStore.test.ts`/`push.test.ts` зелёные. `tsc -b` + `vite build`.
Прод-проба под ROLLBACK подтвердила FK-репро (C) до фикса; прод-схема не изменена.
