# 0003. Renewal idempotency-guard — сверка GET /v3/payments до создания платежа

- Статус: accepted
- Дата: 2026-07-10
- Связано: находка N10, PR #65 (merge `49ec66d`), ветка `wave-3-hardening`, модуль `_shared/renewal-idempotency.ts`

## Контекст

`renew-subscription` создаёт платёж автопродления через `POST /v3/payments` с
детерминированным `Idempotence-Key` = хеш(`user_id` + `valid_until` +
`attempt_no`). Повтор с теми же аргументами ЮKassa дедуплицирует на своей
стороне — вернёт тот же платёж, а не создаст второй.

Но `Idempotence-Key` защищает только от повтора с ТЕМИ ЖЕ аргументами. Есть окно,
где ключ меняется, а платёж по факту уже создан:

- Предыдущий `POST /v3/payments` ушёл в ЮKassa и там создал платёж, но ответ
  оборвался таймаутом — до нас не дошёл.
- Значит `last_renewal_attempt_at` не выставился (мы не узнали об успехе), а
  `attempt_no` на следующем тике cron увеличится.
- Другой `attempt_no` → другой `Idempotence-Key` → ЮKassa НЕ распознает это как
  повтор → создаст ВТОРОЙ платёж по тому же продлению = двойное списание.

## Решение

Добавлен страховочный слой ПЕРЕД созданием нового платежа: сверка с реальным
состоянием в ЮKassa через `GET /v3/payments`.

- Перед `POST /v3/payments` для кандидата дёргаем
  `GET /v3/payments?created_at.gte=<окно>&limit=100` (Basic-credentials магазина).
- `_shared/renewal-idempotency.ts::selectActiveRenewalPayment` фильтрует ответ:
  ищет платёж этого же `user_id` (`metadata.user_id`) с `metadata.renewal` in
  (`'true'`,`'1'`) и статусом из активного набора
  (`pending` / `waiting_for_capture` / `succeeded`).
- Если такой платёж найден — новый НЕ создаём (`continue`, лог
  `existing in-flight renewal payment found via GET /v3/payments (N10 dedup)`);
  итог доведёт `payment-webhook` по вебхуку этого платежа.

**Fail-open по осознанному выбору.** Если `GET /v3/payments` вернул не-2xx,
невалидный body или бросил исключение — `selectActiveRenewalPayment`-путь
возвращает `null`, и cron продолжает создавать платёж. Обоснование: полностью
блокировать автопродление из-за сбоя вспомогательного GET-запроса хуже, чем
редкий риск дубля, который в штатном случае всё равно отсекается
`Idempotence-Key`. Guard — это дополнительный слой на нештатное окно (оборванный
таймаут), а не единственная линия защиты.

Модуль `_shared/renewal-idempotency.ts` намеренно без Deno-зависимостей — покрыт
обычным vitest (чистая функция над списком платежей).

## Последствия

Плюсы: закрыто окно двойного списания при оборванном таймауте предыдущей
попытки; логика вынесена в тестируемую чистую функцию; guard не зависит от
корректности локального счётчика `attempt_no`.

Минусы: дополнительный `GET /v3/payments` на каждого кандидата продления (стоит
одного сетевого запроса на тик). Окно поиска ограничено `created_at.gte` — если
активный платёж старше окна, он не будет найден (на практике окно перекрывает
интервал между тиками cron с запасом). Fail-open означает, что при недоступности
ЮKassa GET защита временно сводится к одному `Idempotence-Key`.

Применение на прод: требует деплоя `renew-subscription` v10 (файлы `index.ts` +
`../_shared/pricing.ts` + `../_shared/renewal-idempotency.ts`, `verify_jwt=false`).
На момент записи ADR деплой не выполнен — код в `develop`.
