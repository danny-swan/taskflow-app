# 0010. Авто-обнаружение и восстановление битой локальной SQLite при старте

- Статус: accepted
- Дата: 2026-07-22
- Связано: находка F16 (эпик «Пространства», roadmap §7.16), ветка `feat/workspaces`,
  [ADR 0009](0009-accept-invite-upsert-membership.md) (раздел «Что осталось»,
  где F16 впервые анонсирован); клиентский фикс, **без серверных миграций**.

## Контекст

Часть пользователей теряла shared-пространства после рестарта приложения:
сайдбар показывал только personal-ws, хотя пользователь точно состоял в
shared-пространстве и раньше видел его. Скриншоты консоли показали два
типовых сообщения драйвера sql.js:

```
Error: database disk image is malformed
  at ... (code 11)
```

```
Error: UNIQUE constraint failed: workspaces.uuid
  at ... (code 2067)
```

а также аналогичные `code: 2067` на `workspace_members.uuid` и
`task_activity_log.uuid`.

**Корень.** Локальная SQLite (веб-режим: sql.js-инстанс, сериализованный в
LocalStorage под ключом `taskflow.sqlite.v1`) физически повреждается —
`PRAGMA integrity_check` на такой базе возвращает не `ok`, либо сам
конструктор/exec бросает `SQLITE_CORRUPT` (code 11, «database disk image is
malformed»). Повреждение затрагивает в первую очередь **индексы**: данные в
таблице физически на месте, но уникальный индекс на `uuid` расходится с
данными. В результате `applyCloudRow*` (см. `src/lib/sync/pull.ts`) выполняет
`SELECT ... WHERE uuid=?` для примения входящей строки из pull — индекс не
находит совпадение (он битый), applier решает, что строки нет, и уходит в
ветку `INSERT`. Но физический уникальный индекс **всё равно видит**
существующую строку с тем же `uuid` → `INSERT` ловит `UNIQUE constraint
failed` (code 2067) → applier бросает исключение.

Снаружи для пользователя это выглядит так: pull к Supabase возвращает `200
OK` (сеть в порядке, RLS не при чём), но ничего не персистится локально —
applier каждой строки падает молча (ошибка тонет в `firstError`, sync
формально «завершается»). На **следующем** старте приложения
`prunePhantomWorkspaces` (`src/lib/sync/pull.ts`, ~строка 897 до этого
фикса) читает локальную (битую, пустую от свежих строк) таблицу
`workspace_members`, не находит live-membership для shared-ws и **удаляет
сам workspace** из локальной таблицы `workspaces` как «фантомный» — сайдбар
пустеет уже окончательно, до следующего полного pull с нуля.

Итог: pull «работает», но каждый цикл на битой базе усугубляет ситуацию,
а prune довершает разрушение локального состояния.

## Решение

Три взаимосвязанные точки защиты:

**1. Детект при старте, до применения любых данных.**
`detectAndRecoverCorruption()` (новая экспортируемая функция, `src/lib/db.ts`)
вызывается в самом начале `initDb()` — **до** `ensureSchema`/`migrate`/
`hydrate`, чтобы миграции и сид не работали поверх мусора и не маскировали
проблему до следующего рестарта.

- **Веб-ветка:** читает сериализованный блоб из `localStorage[STORAGE_KEY]`,
  пытается создать `new SQL.Database(bytes)` и выполнить
  `PRAGMA integrity_check`. Повреждение = конструктор бросил, либо
  integrity_check вернул значение, отличное от `'ok'`, либо exec бросил с
  сигнатурой `SQLITE_CORRUPT`/`code 11`/`database disk image is malformed`.
  **(F16 escalation, добавлено по живой диагностике 22.07.2026):**
  integrity_check иногда возвращает `'ok'`, пока порча ограничена
  рассинхроном одного uuid-индекса с таблицей (точно это было видно в
  прод-логах пользователя — code 2067 появлялся раньше и чаще, чем code 11).
  После успешного integrity_check веб-ветка дополнительно вставляет/удаляет
  строку во временную таблицу с UNIQUE-индексом (`__corruption_probe`) — любое
  исключение на этой тривиальной операции (в т.ч. обычный `UNIQUE constraint
  failed`) трактуется как corruption.
- **Tauri-ветка:** аналогично через `getTauriDb()` +
  `d.select('PRAGMA integrity_check')` на нативном SQLite-соединении; ошибка
  получения самого соединения тоже считается признаком порчи. (UNIQUE-probe
  сейчас реализован только для веб-ветки; для Tauri — см. «что осталось»).
- При обнаружении: `console.error('[db][corruption] detected, will reset:',
  reason)`, веб — `localStorage.removeItem(STORAGE_KEY/STORAGE_KEY_TS)`,
  Tauri — `DROP TABLE IF EXISTS` по всем известным локальным таблицам; если
  и это не удаётся — ставится флаг `window.__taskflow_corruption_unrecoverable`
  (расширение сверх исходного требования — для UI, который в будущем может
  попросить пользователя переустановить приложение). В обоих случаях
  выставляется `window.__taskflow_corruption_recovered = reason`.
  `bound_user_id` живёт в SQLite-таблице `settings` (не в LocalStorage как
  отдельный ключ — см. `src/lib/snapshots.ts`), поэтому при сбросе всего
  локального хранилища он естественным образом обнуляется вместе с остальной
  базой; следующий успешный `syncNow()` привязывает базу к аккаунту заново.

**2. Защита самого pull от runtime-порчи.**
Каждый `applyCloudRow*` в `src/lib/sync/pull.ts` теперь оборачивается
`withCorruptionGuard(tableName, fn)`. Если внутри аппликатора вылетает
исключение с сигнатурой порчи SQLite (та же проверка: «database disk image is
malformed» / `code: 11` / `SQLITE_CORRUPT` — это покрывает и code-2067
UNIQUE-конфликты, чьё сообщение обычно сопровождается или коррелирует с этими
сигнатурами при уже битых индексах), guard поднимает флаг
`window.__taskflow_corruption_recovered = 'runtime_during_pull'` и
перебрасывает **новый** `SqliteCorruptError` (тот же файл, по образцу
существующего `DeferRowError`). `pullTable` ловит `SqliteCorruptError`
отдельно от обычных ошибок строк: не двигает курсор, помечает
`result.corruption = true` и прерывает цикл по этой таблице. `pullSpecPaged`
и `pullAll` проверяют `corruption` после каждого шага и **не вызывают**
`prunePhantomWorkspaces`, возвращая `{..., corruption: true}` вместо обычного
результата. Дополнительно сама `prunePhantomWorkspaces` получила защитный
guard в начале тела: если `window.__taskflow_corruption_recovered` выставлен
в текущем цикле — функция сразу возвращается без действий (комментарий в
коде: «нельзя чистить фантомы сразу после того, как база была битой»).

**2.1. Эскалация UNIQUE-конфликтов до corruption (добавлено 22.07.2026, по живой
диагностике на клиенте).** Строгий инвариант: серверные
`sync_workspace_members` уникальны по `(workspace_id, user_id)` (гарантирует
уникальный индекс и реконсиль `accept_invite` — см. F15/[ADR
0009](0009-accept-invite-upsert-membership.md)). Серверных дубликатов по этой
паре НЕТ — проверено через `execute_sql` на прод-базе. Следовательно, `UNIQUE
constraint failed` на этом индексе при `INSERT`-пути аппликатора (т.е. когда
`SELECT WHERE uuid=?` не нашёл локальную строку и applier решил вставить новую) —
это не бизнес-коллизия, а сигнал о том, что локальный uuid-индекс рассинхронизирован
с таблицей (строка с таким (ws, user) уже есть физически, но индекс на uuid её не
нашёл). `isSqliteCorruptionMessage()` (`src/lib/sync/pull.ts`) расширен
четырьмя добавочными сигнатурами: `workspace_members.workspace_id,
workspace_members.user_id`, `workspaces.uuid`, `workspace_members.uuid`,
`task_activity_log.uuid`. Все четыре наблюдались в реальных логах
пользователя как предвестники последующего `code 11 malformed` (2067
наблюдался первым, затем sql.js каскадно пришёл к malformed — см. роадмап
§7.16).

**3. Восстановление на уровне UI/оркестратора.**
`syncNow()` (`src/lib/sync/index.ts`) после каждого из двух вызовов
`pullAll` проверяет `result.corruption`. Если флаг выставлен — показывает
тост («Локальная база была повреждена, восстановление...») и вызывает
`window.location.reload()` (веб), **не выполняя push** в этом цикле
(push сразу после битой базы опасен — может улететь мусор из ещё не
восстановленного локального состояния). `App.tsx` независимо, при монте,
проверяет `window.__taskflow_corruption_recovered` (на случай, если флаг
выставлен ещё в `initDb()` до того, как приложение успело отрисоваться) и
показывает пользователю мягкое сообщение о восстановлении данных из облака,
затем удаляет флаг.

## Альтернативы — рассмотрены и отклонены

- **A) Принудительный export/import snapshot при каждом закрытии
  приложения.** Периодически сохранять «известно валидный» снапшот и
  откатываться на него при обнаружении порчи. Отклонено: пользователь может
  не заметить момент порчи и продолжать работать поверх битой базы много
  циклов — снапшот тоже успеет устареть или сам окажется битым (если порча
  происходит на серилизации, а не на чтении). Не решает корневую причину
  (индексы), только сдвигает симптом.
- **B) Точечное восстановление битых индексов (`REINDEX`).** Технически
  возможно для некоторых типов повреждения индекса, но не гарантирует
  восстановление при физической порче страниц (code 11) — `REINDEX` на такой
  базе сам может упасть. Более сложная логика с непредсказуемым покрытием
  случаев ради экономии на пересинке из облака, которая для Pro-плана и так
  является источником истины.
- **C) Тихая деградация без auto-reset (просто логировать и продолжать).**
  Отклонено — это ровно текущее (проблемное) поведение: ошибки тонут в
  `firstError`, prune удаляет фантомные ws, пользователь теряет данные без
  объяснения. Не решает проблему, только скрывает её лучше.

Выбор — **integrity_check при каждом старте + автосброс локального
хранилища + пропуск prune на цикле восстановления + тост-уведомление**,
минимальная поверхность изменений, использует уже существующий контракт
«облако — источник истины» для Pro/trial.

## Тесты

- `src/lib/db.corruption.test.ts` (vitest): 4 сценария —
  (a) чистый `localStorage` → `{recovered:false}`, storage не тронут;
  (b) мусорные байты вместо валидной SQLite (`JSON.stringify([1,2,3,4,5])`)
  → `{recovered:true}`, storage очищен;
  (c) валидная пустая база (настоящий sql.js-экспорт через `initDb()`) →
  `{recovered:false}`;
  **(d) добавлено 22.07.2026, F16 escalation:** побитовое повреждение
  валидного sql.js-экспорта (XOR 0xFF по одной из нескольких пробных позиций
  в диапазоне байт 100-200) → `{recovered:true}` хотя бы на одной из позиций
  (тест мягкий по дизайну — побитовая порча вне 100-байтного заголовка не
  гарантированно детектируется на каждой конкретной позиции, но хотя бы одна из
  пяти пробных точек должна сработать через integrity_check или UNIQUE-probe;
  при запуске детект подтвердился через `integrity_check` (`SQLITE_CORRUPT:
  database disk image is malformed`).
  Покрывает только веб-ветку — тесты гоняются в vitest+jsdom, где
  `IS_TAURI` (вычисляется из `window.__TAURI_INTERNALS__`) всегда `false`;
  Tauri-ветка (нативный `@tauri-apps/plugin-sql`) не воспроизводима в этом
  окружении и проверяется вручную/в e2e.

## Последствия

- Данные, ещё не отправленные в облако (строки в `sync_outbox`, накопленные
  локально до момента обнаружения порчи), **будут потеряны** при автосбросе
  локального хранилища — облако не содержит того, что не успело запушиться.
  Это принято как компромисс: для Pro/trial-плана облако уже является
  источником истины (см. существующий контракт `syncNow`), а альтернатива
  (продолжать работать на битой базе) достоверно приводит к потере данных
  без явного уведомления и к разрушению shared-workspaces через
  `prunePhantomWorkspaces`. Free-план не затрагивается сетевой веткой синка,
  но локальный bootstrap тоже пересеется с нуля при обнаружении порчи.
- `bound_user_id` неявно обнуляется вместе со сбросом хранилища —
  не отдельная логика, а побочный эффект полного сброса блоба. Задокументировано
  здесь для будущих читателей кода, которые могут искать явный
  `localStorage.removeItem('bound_user_id')` и не найти его.
- Восстановление зависит от последующего успешного `syncNow()` — если сеть
  недоступна сразу после обнаружения порчи, пользователь временно увидит
  пустое (но не битое и не теряющее данные) состояние вместо мгновенного
  восстановления shared-workspaces.

## Что осталось (не в этом PR)

- `window.__taskflow_corruption_unrecoverable` (Tauri, когда даже
  `DROP TABLE IF EXISTS` не помогает) сейчас только выставляет флаг — UI,
  который предлагает пользователю переустановить приложение, не реализован
  в этом PR и не входит в scope F16.
- **UNIQUE-probe (после F16 escalation) реализован только для веб-ветки.**
  Tauri-ветка `detectAndRecoverCorruption()` пока полагается только на
  `PRAGMA integrity_check`, без аналогичной вставки/удаления временной таблицы через
  `@tauri-apps/plugin-sql`. Не было частью текущего патча (нет прод-диагностики с
  Tauri-клиента для этого случая) — стоит добавить при появлении аналогичных
  симптомов на desktop.
- Сигнатуры `isSqliteCorruptionMessage()` завязаны на точные имена таблиц/колонок
  (`workspace_members.workspace_id, workspace_members.user_id` и т.д.) — если
  схема будет переименована или появятся новые UNIQUE-индексы в sync-таблицах,
  эти регексы надо будет синхронизировать вручную.
- Расследование `sync_devices RLS violation` (~30/час, впервые упомянуто в
  [ADR 0009](0009-accept-invite-upsert-membership.md)) остаётся открытым —
  вероятная связь с битым `client_id` на повреждённых базах не подтверждена
  в этом фиксе.

## Обновление (F17, 22.07.2026, ADR 0011)

**F16-эскалация 2067→corruption пересмотрена и откачена.** Живая диагностика
под ролью приглашённого (прод-SQL) показала, что `UNIQUE constraint failed:
workspace_members.workspace_id, workspace_members.user_id` (code 2067) —
**НЕ признак порчи** локальной SQLite: `integrity_check` возвращает `ok`,
строки физически на месте, индекс не рассинхронизирован. Настоящая причина —
рассинхрон **идентичности** строки членства: при accept-invite клиент создаёт
локальную строку со своим случайным `uuidv7`, а сервер (`accept_invite`,
F15/[ADR 0009](0009-accept-invite-upsert-membership.md), миграция 0040) хранит
свой канонический uuid для той же пары `(workspace_id, user_id)`. Матчинг
`applyCloudRowMembers` только по uuid промахивался → INSERT → 2067 на
не-partial индексе `idx_workspace_members_ws_user`.

Следствия для этого ADR:

- Четыре UNIQUE-паттерна (`workspace_members.workspace_id, user_id`,
  `workspaces.uuid`, `workspace_members.uuid`, `task_activity_log.uuid`)
  **удалены** из `isSqliteCorruptionMessage()` (`src/lib/sync/pull.ts`).
  Теперь эта функция ловит только настоящую порчу: `database disk image is
  malformed`, `code: 11`, `SQLITE_CORRUPT`. Раздел «Решение → 2.1» и
  соответствующие упоминания в разделе «Решение → 1» (F16 escalation про
  четыре сигнатуры) считать **отменёнными** — оставлены в тексте только как
  исторический контекст.
- `__corruption_probe` (веб-ветка `detectAndRecoverCorruption`, `src/lib/db.ts`)
  **сохранён** — он тестирует реальную порчу индекса на СВОЕЙ изолированной
  temp-таблице и не связан с бизнес-коллизией 2067 на `workspace_members`
  (в код добавлен поясняющий комментарий).
- Сама коллизия перенесена из «сброс всей локальной БД» в точечный
  fallback-матчинг по `(workspace_id, user_id)` с переклейкой uuid на
  серверный — см. [ADR 0011](0011-membership-uuid-mismatch-reconcile.md)
  и roadmap §7.17.

Всё остальное в этом ADR (детект настоящей порчи по `integrity_check`,
`withCorruptionGuard` на сигнатуры malformed/code 11/SQLITE_CORRUPT,
prune-skip, тост + reload) остаётся в силе.
