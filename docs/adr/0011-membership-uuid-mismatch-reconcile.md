# 0011. Реконсиль uuid членства при рассинхроне local↔server (fallback-матчинг по натуральному ключу)

- Статус: accepted
- Дата: 2026-07-22
- Связано: находка F17 (эпик «Пространства», roadmap §7.17), ветка `feat/workspaces`;
  [ADR 0008](0008-workspace-members-full-pull-leave-refresh.md) (F14 — двухфазный
  pull членства), [ADR 0009](0009-accept-invite-upsert-membership.md) (F15 —
  `accept_invite` upsert, стабильный серверный uuid),
  [ADR 0010](0010-sqlite-corruption-auto-recovery.md) (F16 — авто-восстановление
  битой SQLite; **частично отменяет** его эскалацию 2067→corruption, см. ниже);
  клиентский фикс, **без серверных миграций**.

## Контекст

Часть пользователей теряла shared-пространства после рестарта приложения:
сразу после принятия приглашения пространство было видно, но после перезапуска
сайдбар показывал только personal-ws. В консоли F12:

```
Error: UNIQUE constraint failed: workspace_members.workspace_id, workspace_members.user_id
  at ... (code 2067)
```

Изначально (F16/[ADR 0010](0010-sqlite-corruption-auto-recovery.md)) этот 2067
был ошибочно классифицирован как признак **порчи** локальной SQLite: гипотеза
была, что битый uuid-индекс не находит существующую строку, applier уходит в
INSERT, а физический уникальный индекс всё равно ловит дубликат. По этой
гипотезе F16 расширил `isSqliteCorruptionMessage()` четырьмя UNIQUE-паттернами и
на любой такой 2067 сбрасывал всю локальную базу.

**Живая диагностика под ролью приглашённого (прод-SQL) опровергла гипотезу
порчи:** `PRAGMA integrity_check` возвращает `ok`, строки физически на месте,
индекс с данными согласован. База **не битая**.

**Корень (доказан прод-SQL).** Рассинхрон **идентичности** строки членства:

- При accept-invite клиент создаёт локальную строку `workspace_members` со
  СВОИМ случайным `uuidv7` (например `wsm_77cf64e183584081b1803e68b83f9b30`).
- Сервер в `accept_invite` (F15/[ADR 0009](0009-accept-invite-upsert-membership.md),
  миграция 0040) хранит СВОЙ канонический uuid для той же пары
  `(workspace_id, user_id)` — при повторном invite после leave он ещё и
  реактивирует существующую серверную строку, сохраняя её uuid.
- `applyCloudRowMembers` (`src/lib/sync/pull.ts`) матчил локальную строку
  ТОЛЬКО по uuid: `SELECT ... WHERE uuid = row.id`. Серверный uuid ≠ локальный
  случайный → matcher промахивался → applier уходил в ветку `INSERT`.
- Локальный уникальный индекс `idx_workspace_members_ws_user` на
  `(workspace_id, user_id)` **НЕ partial** (срабатывает даже на soft-deleted
  строках) → пара уже занята → `INSERT` падал с 2067.
- Ошибка тонула в `firstError`, pull формально завершался, ничего не
  персистилось. На следующем старте `prunePhantomWorkspaces` (pull.ts) не
  находил «живого» членства для shared-ws и удалял его как фантомное —
  сайдбар пустел окончательно.

Механизм разрушения тот же, что описан в F16, но **причина иная**: не порча
индекса, а расхождение локального и серверного uuid для одной и той же строки
членства. Серверных дубликатов по `(workspace_id, user_id)` нет (гарантируют
уникальный индекс `sync_workspace_members` и реконсиль `accept_invite` —
F15) — значит 2067 на этой паре при INSERT-пути аппликатора всегда означает
«у меня локально уже есть эта строка, но под другим uuid».

## Решение

Две связанные точки:

**1. Fallback-матчинг в `applyCloudRowMembers` (`src/lib/sync/pull.ts`).**
Матчинг стал двухступенчатым:

1. **По uuid** (`byUuid = SELECT id, updated_at WHERE uuid = row.id`) — если
   найдена, поведение прежнее: LWW по `updated_at`, `UPDATE ... WHERE uuid`.
2. **По uuid промах → fallback по натуральному ключу**
   (`byPair = SELECT id, uuid, updated_at WHERE workspace_id=? AND user_id=?`).
   Если строка найдена — это тот же membership с локальным uuid ≠ серверный.
   НЕ вставляем (иначе 2067). **ВСЕГДА выставляем `uuid = row.id`** — серверный
   uuid является канонической идентичностью членства. Прочие поля (role,
   invited_by, joined_at, deleted_at, version, client_id, workspace_id,
   user_id, updated_at) — по LWW: если `row.updated_at > byPair.updated_at` —
   обновляем всё + uuid; иначе обновляем ТОЛЬКО uuid. `UPDATE ... WHERE id = byPair.id`.
   - **Edge case:** перед переклейкой проверяем, не занят ли серверный uuid
     ДРУГОЙ локальной строкой (`SELECT id WHERE uuid = row.id AND id <> byPair.id`).
     Если да (например, остаток от прошлой неудачной переклейки) — удаляем
     дубликат-byPair (`DELETE WHERE id = byPair.id`), пишем `logger.warn`, и
     применяем LWW к строке-владельцу uuid.
3. **Ни uuid, ни пара не найдены** → `INSERT`, как раньше.

`applyCloudRowWorkspaces` проверен и **fallback не требует**: локальная таблица
`workspaces` имеет только partial-индекс `idx_workspaces_uuid`
(`WHERE uuid IS NOT NULL`), натурального UNIQUE-ключа нет, поэтому симметричной
2067-коллизии там не возникает (id пространства детерминирован — `ws_<uid>`
для personal, серверный uuid для shared — и приходит одинаковым с обеих сторон).

**2. Откат F16-эскалации 2067→corruption.**
Из `isSqliteCorruptionMessage()` (`src/lib/sync/pull.ts`) удалены четыре
UNIQUE-паттерна (`workspace_members.workspace_id, user_id`, `workspaces.uuid`,
`workspace_members.uuid`, `task_activity_log.uuid`). Осталась только сигнатура
настоящей порчи: `database disk image is malformed`, `code: 11`,
`SQLITE_CORRUPT`. Теперь штатный 2067 → `isSqliteCorruptionMessage = false` →
`withCorruptionGuard` перебрасывает обычную ошибку строки (в `firstError`),
локальное хранилище **НЕ сбрасывается**. После фикса матчинга этот 2067 для
пары `(workspace_id, user_id)` вообще не должен возникать — fallback ловит
строку раньше, чем applier дойдёт до INSERT.

`__corruption_probe` в `detectAndRecoverCorruption()` (`src/lib/db.ts`,
веб-ветка) **сохранён** — он тестирует реальную порчу индекса на СВОЕЙ
изолированной temp-таблице и не связан с бизнес-коллизией 2067 на
`workspace_members`. В код добавлен поясняющий комментарий, чтобы будущие
читатели не приняли probe за часть F16-эскалации.

## Альтернативы — рассмотрены и отклонены

- **A) Оставить F16-трактовку (2067 = corruption, сброс базы).** Отклонено:
  это лечит симптом ценой потери всех незапушенных локальных данных при каждой
  штатной коллизии uuid. Диагностика доказала, что база не битая — сброс здесь
  избыточен и вреден.
- **B) Матчить членство только по натуральному ключу `(workspace_id, user_id)`,
  игнорируя uuid.** Отклонено: uuid нужен для outbox/push (обратная отправка
  строки) и для строк, где натуральный ключ может быть неполным (в теории
  `user_id` может быть null у pending-инвайта до accept). Двухступенчатая схема
  (uuid → пара) сохраняет обратную совместимость и покрывает оба случая.
- **C) Сделать локальный `idx_workspace_members_ws_user` partial
  (`WHERE deleted_at IS NULL`), чтобы soft-deleted строки не конфликтовали.**
  Отклонено как недостаточное: основной кейс (accept-invite) конфликтует с
  ЖИВОЙ строкой (deleted_at IS NULL), а не с soft-deleted — partial-индекс его
  не спасёт. Плюс потребовал бы миграцию схемы, а фикс задумывался клиентским
  без DDL.

Выбор — **fallback-матчинг по натуральному ключу с переклейкой uuid на
серверный + откат ошибочной 2067-эскалации F16**: минимальная поверхность
изменений, без миграций, устраняет первопричину (рассинхрон идентичности), а не
симптом.

## Последствия

- 2067 на `workspace_members(workspace_id, user_id)` больше НЕ приводит к
  сбросу локальной SQLite. Пользователи, у которых shared-ws пропадали после
  рестарта из-за этого, после фикса видят пространство стабильно — локальная
  строка членства «доклеивается» серверным uuid при первом же pull.
- Незапушенные локальные данные больше не теряются на этой коллизии (в отличие
  от F16-поведения, где любой 2067 стирал базу).
- Матчинг членства теперь зависит от натурального ключа `(workspace_id,
  user_id)` — если серверная модель когда-либо допустит несколько строк
  членства на одну пару (сейчас запрещено уникальным индексом
  `sync_workspace_members` + `accept_invite`), fallback выберет первую и
  переклеит на неё uuid; такое изменение серверной модели потребует пересмотра
  этой логики.
- `__corruption_probe` (F16) остаётся активным и продолжает ловить настоящую
  порчу движка/страниц независимо от F17.

## Что осталось (не в этом PR)

- `applyCloudRowWorkspaces` fallback не получил (обоснованно — нет натурального
  UNIQUE). Если в `workspaces` появится натуральный уникальный ключ, симметричную
  логику надо будет добавить.
- Fallback реализован для `workspace_members`; другие sync-таблицы с локальным
  uuid ≠ серверный (если такие появятся) потребуют аналогичного матчинга.
- Расследование `sync_devices RLS violation` (~30/час, из
  [ADR 0009](0009-accept-invite-upsert-membership.md)/[ADR 0010](0010-sqlite-corruption-auto-recovery.md))
  остаётся открытым — к F17 напрямую не относится.
- Отдельный `Uncaught TypeError: Cannot create property '_id' on number` из
  минифицированного sync-стека (наблюдался в проде) в этом PR не воспроизведён
  и не устранён — исходного (не-минифицированного) присваивания свойства на
  число в sync-коде не найдено; вероятнее всего `_id` — mangled-имя свойства,
  которое сторонняя библиотека (Sentry/PostgREST-обёртка) пытается навесить на
  примитив, пришедший туда, где ожидался объект. Требует прод-сорсмапа для
  точной локализации; к блокеру F17 не относится.
