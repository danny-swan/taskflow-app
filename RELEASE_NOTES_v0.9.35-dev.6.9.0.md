# TaskFlow v0.9.35-dev.6.9.0 — Release Notes

**Дата:** 08.07.2026
**Тип:** Feature — изоляция локальной базы по аккаунту + снимки (safety net) для смены аккаунта
**Ветка:** develop

---

## TL;DR

Локальная база TaskFlow — это **один файл на устройство** (`data.db`), а не на аккаунт. Раньше, если на одном устройстве выйти из аккаунта A и войти под аккаунтом B (например, у которого своя база на другом устройстве), синхронизация могла молча смешать или перезаписать данные — всё было завязано на облако.

Теперь при **смене аккаунта** приложение это распознаёт и показывает модальное окно с явным выбором, что делать с текущими локальными данными:

- **Загрузить облачные** — локальные данные очищаются, скачиваются задачи нового аккаунта из облака.
- **Оставить локальные** — текущие задачи остаются и записываются в облако нового аккаунта.
- **Объединить** — локальные и облачные задачи сливаются вместе.

**Ключевая гарантия:** перед **любым** из этих действий автоматически создаётся **снимок** текущей локальной базы. Даже если выбрать «Загрузить облачные», старые данные не теряются — их всегда можно вернуть из снимка в **Настройки → Синхронизация → Снимки локальной базы**. Ничего не завязано только на облаке.

---

## Проблема

- Локальная база = один файл на устройство (`data.db`), не привязана к конкретному аккаунту — корень бага.
- При входе под другим аккаунтом на том же устройстве sync (pull→push→pull, LWW по `updated_at`, склейка по UUID) применялся к чужой локальной базе. В худшем случае несинхронизированные локальные задачи одного аккаунта могли уйти в облако другого или быть перезаписаны облачными.
- Не было точки, где пользователь мог бы осознанно решить судьбу данных, и не было офлайн-страховки (всё завязано на облако).

---

## Решение (архитектура)

### Привязка базы к аккаунту (`bound_user_id`)

- В `settings` хранится ключ `bound_user_id` — id аккаунта, которому «принадлежит» текущая локальная база.
- **Отсутствие** ключа = «база не привязана» (первый вход, либо старая база до migration v8). Гейт в этом случае не показываем — привязку выставит первый успешный sync.
- После успешного sync (`status: 'synced'`) `bound_user_id` записывается/обновляется (best-effort, не критично для sync).
- `checkAccountBinding(sessionUserId)`:
  - не привязана → `mismatch = false`;
  - `bound_user_id === sessionUserId` → `mismatch = false` (тот же аккаунт);
  - `bound_user_id !== sessionUserId` → `mismatch = true` (**смена аккаунта → показать гейт**).

### Снимки локальной базы (safety net)

Два бэкенда (как и в `db.ts`):

- **Tauri (desktop, приоритет):** полноценные бинарные копии файла `data.db` в подпапке `snapshots/` рядом с базой (через Rust-команды). Переживают перезапуск, не зависят от localStorage.
- **Web (браузер):** снимок = JSON-дамп (`buildBackup`) в localStorage. Для разработки/будущей web-версии; ограничен объёмом localStorage — UI показывает мягкое предупреждение (`isWebSnapshotLimited()`).

Метаданные (id, метка, дата, размер, платформа, привязка, кол-во задач) хранятся в `settings.snapshot_registry_v1` как JSON-массив — кросс-платформенно, без новой таблицы.

**Ротация:** держим не более **5** снимков. При создании нового самый старый тихо удаляется (и файл/payload, и запись реестра).

## Что изменено

### 1. Миграция `migrations.ts` — v8

- `INSERT OR IGNORE` в `settings`: `snapshot_registry_v1 = '[]'` (пустой реестр).
- `bound_user_id` НЕ создаётся заранее — отсутствие ключа означает «не привязана».
- Идемпотентно, `TARGET_VERSION` авто-выведен из длины массива миграций.

### 2. Rust `src-tauri/src/main.rs` — 4 команды снимков

- `snapshot_db(label) → SnapshotFileInfo` — бинарная копия `data.db` в `snapshots/snapshot_<epoch_ms>_<safe_label>.db`.
- `list_snapshots() → Vec<SnapshotFileInfo>` — перечисление файлов снимков.
- `restore_snapshot(snapshot_path) → String` — восстановление: **сначала** страховочная копия текущей БД, потом замена файла, удаление WAL/SHM. Возвращает путь страховочной копии. Требует перезапуск.
- `delete_snapshot(snapshot_path)` — удаление файла снимка.
- Защита от path-traversal через `canonicalize` + `starts_with` (только внутри папки снимков).
- Helpers: `resolve_db_path`, `snapshots_dir`, `sanitize_label`; struct `SnapshotFileInfo` (Serialize). Все 4 команды в `generate_handler!`.

### 3. `src/lib/snapshots.ts` (новый) — единый API снимков

- `MAX_SNAPSHOTS = 5`, реестр в `snapshot_registry_v1`, web-payload префикс `taskflow.snapshot.`.
- Экспорт: `readRegistry`, `getBoundUserId`, `setBoundUserId`, `isWebSnapshotLimited`, `checkAccountBinding`, `createSnapshot`, `rotate`, `listSnapshots`, `deleteSnapshot`, `restoreSnapshot`.
- Tauri через `invoke`, Web через localStorage + `buildBackup`/`applyBackup`.

### 4. `src/lib/db.ts` — `clearUserData()`

- Очищает локальные данные без re-seed: `overdue_events`, `tasks`, `task_templates`, `tags`, `statuses`, `sync_outbox`, `sqlite_sequence`, `sync_last_pulled_%` из `settings`.
- **Сохраняет:** `client_id`, `snapshot_registry_v1`, `bound_user_id` (снятие — на усмотрение вызывающего), UI-настройки.
- Используется для варианта «Загрузить облачные»: после очистки следующий sync подтянет данные нового аккаунта.

### 5. `src/lib/sync/index.ts` — запись привязки

- После `status: 'synced'` записывается `bound_user_id`, если изменился (try/catch, не критично для sync).

### 6. `src/components/AccountSwitchGate.tsx` (новый) — модалка выбора

- Детект смены аккаунта через `checkAccountBinding` на изменение `sessionUserId`; при `mismatch` открывается (обязательный выбор, не закрывается по клику вне).
- Три варианта: `cloud` (`clearUserData` + `setBoundUserId(null)` + syncNow), `local` (`setBoundUserId(session)` + syncNow), `merge` (`setBoundUserId(session)` + syncNow).
- **ВСЕГДА** `createSnapshot('before_account_switch')` перед действием.
- Дисклеймер для «Оставить локальные»: «Локальные задачи будут записаны в облако аккаунта. Если эти задачи принадлежали другому аккаунту и не были синхронизированы в его облако, в облаке того аккаунта они не появятся — но локальная копия сохранена в снимке и её можно восстановить.»
- Локальный L10N (ru/en), защита от повторного открытия.

### 7. `src/App.tsx`

- Смонтирован `AccountSwitchGate` рядом с `PasswordResetModal`; не рендерится при E2E-bypass (`?e2e=1`).

### 8. `src/pages/Settings.tsx` — UI снимков в SyncSection

- Блок «Снимки локальной базы»: список (метка, дата, размер, кол-во задач), кнопки «Восстановить», «Удалить», «Создать снимок вручную».
- Восстановление на desktop показывает `ConfirmDialog` перезапуска (`restart_app`).
- Для web — предупреждение `isWebSnapshotLimited()`.

### 9. `src/pages/Help.tsx` — актуализация FAQ

- Убрано устаревшее «cloud sync на roadmap / task content не загружается в облако» (RU + EN).

## Регрессии / риски

- Миграция v8 идемпотентна и аддитивна (только вставка в `settings`) — существующие базы не ломаются.
- `clearUserData()` сохраняет снимки и `client_id` — «Загрузить облачные» обратимо через снимок.
- Rust-восстановление делает страховочную копию ДО замены файла — откат восстановления возможен.
- Path-traversal защита в Rust (canonicalize + starts_with) — restore/delete только внутри `snapshots/`.
- E2E-bypass сохранён — гейт не мешает Playwright-тестам.
- 24 unused индекса на `sync_*` НЕ трогаются (по договорённости — до стабилизации sync).
